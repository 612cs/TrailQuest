# TrailQuest UI 开发规范

为了保持项目视觉风格的一致性和专业感，所有开发人员必须遵循以下 UI 准则：

## 1. 禁止使用 Emoji 表情符号
- **规定**：严禁在任何页面或组件的 UI 文本、图标或装饰中使用系统自带的 Emoji 表情符号（例如：🥾, 🏃, 🌡️）。
- **原因**：Emoji 在不同操作系统（iOS, Android, Windows, macOS）上的显示效果差异巨大，且往往带有过于休闲的风格，不符合项目追求的“高端、专业”视觉定位。
- **替代方案**：请统一使用 `lucide-vue-next` 图标库。

## 2. 组件化原则
- **抽离逻辑**：任何具有独立功能或可复用样式的 UI 区块都应抽离为独立组件。
- **目录规范**：
  - `src/components/`：存放通用基础组件（如 Button, Card, Icon）。
  - `src/views/[ViewName]/components/`：存放仅在特定视图中使用的子组件（可选，视项目规模而定）。
- **保持专注**：主页面（View 文件）应主要负责布局、数据获取和子组件组装，尽量避免包含大量的原始 HTML 和复杂的内联样式。

## 3. 设计美学
- **图标一致性**：使用统一的线条粗细（推荐 `stroke-width="2"`）。
- **色彩规范**：使用项目中定义的 CSS 变量（如 `var(--primary-500)`），避免硬编码颜色值。
- **动效**：利用 `animate-fade-in-up` 等预设动画增强交互感。
- **主题**：所有的新的UI开发都要支持暗黑和白天主题的切换！！！

# 代码提交桂发

- **提交信息**：遵循约定式提交规范（Conventional Commits），如 `feat:`, `fix:`, `docs:`, `refactor:` 等，并且使用中文进行提交。
- **工具优先**：优先使用 GitHub MCP 进行跨系统的直接提交，以确保操作的原子性和可追溯性。

# 代码书写规范

## 1. 数据管理与后端对接预备

- **Mock 规范**：所有 Mock 数据必须统一存放在 `src/mock/` 目录下（如 `mockData.ts`），严禁在组件内直接硬编码大量测试数据，以保持逻辑与数据的分离。
- **接口模拟**：在设计数据结构和 TypeScript 接口时，应尽可能贴合真实的业务模型（如使用 `id` 索引、标准化的时间戳、嵌套的评论结构等），降低后续对接真实 API 的迁移成本。
- **逻辑封装**：数据请求逻辑（包括目前的 Mock 获取）应优先封装在 Pinia Store 或 Service 层中。组件应通过调用这些层提供的方法来获取数据，而不是直接引用 Mock 对象，确保切换到后端接口时只需修改逻辑层代码。
- **类型安全**：强制为所有 API/Mock 返回的数据定义清晰的 `interface` 类型，利用 TypeScript 的强类型检查减少运行时错误。
