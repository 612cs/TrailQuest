# 下一阶段任务拆包与实施建议

这份文档用于承接 `DOCS/handoff/current-progress-handoff-2026-03-16.md` 之后的下一阶段开发安排。目标不是立即开发，而是把接下来的工作拆成 3 个可连续提交的小功能包，保证每一包都能独立落地、独立验证、独立提交。

## 1. 当前前置条件

截至 2026-03-16，当前已经明确的前置条件如下：

- 路线列表、搜索、社区、路线详情主数据已接真实接口
- 评论真实接口已经完成
- 评论图片真实上传、回显、二级/三级回复链路已打通
- 评论相关的历史 `trail_id` 脏数据修复 SQL 已经执行完成
- 评论区当前规则已经稳定为：
  - 每条评论都有 `id`
  - 每条评论都有 `trail_id`
  - 顶级评论 `parent_id = null`
  - 回复评论 `parent_id != null`
  - 页面最多展示三级评论
  - 回复三级评论时归并为三级展示，不再报错
- OSS 上传链路已经稳定，可继续复用到详情页后续能力

换句话说，路线详情页当前已经具备：

- 详情主信息展示
- 路线天气与定位展示
- 评论查看
- 评论发布
- 评论图片上传
- 二级/三级回复

但以下能力仍未闭环：

- 分享按钮
- 点赞/收藏真实交互
- 评论删除
- 评论懒加载
- `@用户` 卡片
- 天气与地理解析后端迁移

## 2. 本阶段明确不做的内容

本轮已经明确暂缓，不纳入下一阶段实现范围：

- 相关路线推荐
- 作者其他路线
- 评论排序切换
- 传统页码分页器
- 评论编辑
- 评论举报
- 评论置顶
- 个人主页扩展项
- 后台管理页面

这样做的原因是先把“路线详情互动闭环”做完整，避免范围继续膨胀。

## 3. 功能包 A：路线详情互动闭环

### 3.1 目标

完善路线详情页的分享按钮，并接入点赞/收藏真实接口和前端状态联动，让详情页从“可查看”升级为“可互动”。

### 3.2 范围

本包只处理路线详情互动，不处理评论增强、不处理天气迁移。

需要完成：

- 路线详情页分享按钮可用
- 路线详情页点赞按钮支持真实读写
- 路线详情页收藏按钮支持真实读写
- 首页、搜索页、社区页、路线详情页的点赞/收藏数与状态联动
- 未登录用户点击点赞/收藏时走现有登录弹窗

### 3.3 后端建议

建议新增或补齐以下接口：

- `POST /api/trails/{id}/like`
- `DELETE /api/trails/{id}/like`
- `POST /api/trails/{id}/favorite`
- `DELETE /api/trails/{id}/favorite`

当前用户对该路线的互动状态，建议二选一：

方案一，推荐：

- 直接并入 `GET /api/trails/{id}` 返回
- 例如增加：
  - `likedByCurrentUser`
  - `favoritedByCurrentUser`

方案二：

- 单独提供 `GET /api/trails/{id}/interaction-status`

推荐优先方案一，原因是详情页进入时本来就会拉详情数据，合并返回更省一次请求。

### 3.4 前端建议

详情页交互建议按以下优先级实现：

- 分享：
  - 优先使用 Web Share API
  - 不支持时降级为复制当前路线链接到剪贴板
  - 成功后给出轻提示
- 点赞/收藏：
  - 允许乐观更新
  - 请求失败必须回滚计数和按钮状态
  - 列表页和详情页尽量共用统一的数据适配逻辑

### 3.5 验收标准

- 点击分享按钮后，移动端可调起系统分享，桌面端至少能复制链接
- 点赞/收藏接口成功后，详情页数值和按钮状态即时更新
- 刷新页面后，当前用户点赞/收藏状态正确恢复
- 失败时状态回滚，不能出现计数错乱
- 首页、搜索页、社区页进入详情后，互动数据保持一致

### 3.6 提交建议

建议单独提交为：

`feat(trail): 完善路线详情分享并接入点赞收藏真实交互`

## 4. 功能包 B：评论区增强包

### 4.1 目标

在现有评论真实链路之上，补齐评论懒加载、评论删除、点击 `@用户` 出现用户卡片。

### 4.2 范围

本包只增强评论区，不处理点赞收藏，不处理天气迁移。

需要完成：

- 评论列表不再一次性整树全量加载
- 评论区支持懒加载更多顶级评论
- 评论只支持“删除”这一种管理动作
- 二级/三级评论中的 `@用户` 可点击并弹出轻量用户卡片

### 4.3 后端建议

评论懒加载不做传统分页器，建议按“顶级评论分批加载”设计。

推荐接口能力：

- `GET /api/trails/{id}/reviews`
  - 增加懒加载参数，例如：
    - `cursor`
    - `limit`
  - 或：
    - `lastId`
    - `pageSize`
  - 文档层面不强制具体字段名，但必须明确“这是加载更多语义，不是页码分页”

删除评论接口：

- `DELETE /api/reviews/{id}`

用户卡片接口：

- `GET /api/users/{id}/card`

建议返回轻量信息：

- `id`
- `username`
- `avatar`
- `avatarMediaUrl`
- `role`
- `joinDate`
- `postCount`
- `savedCount`
- `bio`
- `location`

### 4.4 前端建议

评论区交互建议如下：

- 初次进入详情页时只加载首批顶级评论
- 使用“加载更多”按钮或滚动触底懒加载，不做页码分页器
- 每次追加的是“顶级评论块”，单个顶级评论仍可携带其回复树
- 删除按钮只在当前登录用户自己的评论上显示
- 删除顶级评论时整棵回复树一并移除，以数据库级联删除为准
- `@用户` 点击后弹出轻量用户卡片浮层，不跳完整个人主页

### 4.5 验收标准

- 评论区首屏不再一次性拉全量数据
- 点击“加载更多”或滚动触底后能继续追加评论
- 本人评论可删除，删除后页面即时更新
- 非本人评论不显示删除入口
- 点击二级/三级评论中的 `@用户名`，能看到用户卡片

### 4.6 提交建议

建议单独提交为：

`feat(review): 增强评论懒加载删除与用户卡片交互`

## 5. 功能包 C：天气与地理解析后端迁移

### 5.1 目标

把路线详情页当前前端直连的地理解析与天气查询迁移到后端，统一能力出口，并为后续缓存、鉴权、限流和 key 隐藏做准备。

### 5.2 范围

本包只处理路线详情页的地理与天气能力，不扩展到搜索页、发布页或其它模块。

需要完成：

- 前端不再直接请求高德定位与天气接口
- 后端统一负责路线详情所需的地理解析与天气查询
- 返回路线详情页所需的最小数据结构

### 5.3 后端建议

推荐优先做单一聚合接口：

- `GET /api/trails/{id}/geo-weather`

建议返回字段：

- `center`
- `city`
- `adcode`
- `weather`
- `temperature`
- `humidity`
- `windDirection`
- `windPower`
- `reportTime`

若实现时发现聚合接口过重，可退一步拆成：

- `GET /api/trails/{id}/geo`
- `GET /api/weather?adcode=`

但当前建议优先单接口，减少前端串联请求复杂度。

### 5.4 前端建议

- `TrailDetailView` 改为只请求后端聚合接口
- 地图组件仍可保留现有展示方式，只替换数据来源
- 失败时保持现有降级提示文案
- 不新增复杂重试 UI，先保证稳定替换现有直连逻辑

### 5.5 验收标准

- 路线详情页不再直接调用高德天气与地理解析
- 前端只依赖后端返回的统一结构
- 天气提醒、天气卡片、地图定位仍能正常显示
- 后端接口异常时，前端维持当前降级文案

### 5.6 提交建议

建议单独提交为：

`feat(weather): 将路线详情地理与天气能力迁移到后端`

## 6. 推荐实施顺序与提交建议

推荐顺序如下：

1. 功能包 A：路线详情互动闭环
2. 功能包 B：评论区增强包
3. 功能包 C：天气与地理解析后端迁移

推荐原因：

- A 包最直接补齐用户最容易感知的互动闭环
- B 包建立在现有评论真实链路之上，延续性最好
- C 包属于能力迁移，技术改动面较大，放到最后更稳妥

建议每个功能包都保持：

- 单独开发
- 单独验证
- 单独提交

避免把详情页互动、评论增强、天气迁移混成一个大提交，后续回归和排错会更困难。

## 7. 整体验收清单

下一阶段全部完成后，路线详情页应至少达到以下状态：

- 分享按钮可用
- 点赞/收藏有真实接口与真实状态
- 评论支持懒加载
- 评论支持删除
- `@用户` 可弹出轻量用户卡片
- 天气与地理解析由后端统一提供

同时，以下内容仍然明确不在本阶段：

- 相关路线推荐
- 作者其他路线
- 评论排序切换
- 传统分页器
- 评论编辑 / 举报 / 置顶
- 后台管理

## 8. 一句话结论

下一阶段最适合按 `详情互动闭环 -> 评论区增强 -> 天气后端迁移` 这 3 个小功能包连续推进，每一包都可以独立开发、独立验证、独立提交，不会互相绞在一起。
