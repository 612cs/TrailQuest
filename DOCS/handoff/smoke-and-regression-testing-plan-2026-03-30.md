# TrailQuest 冒烟测试与回归测试方案（2026-03-30）

## 1. 结论摘要

对于 TrailQuest 当前阶段，冒烟测试和回归测试都非常有必要，而且优先级不低。

原因很直接：

- 项目已经不再是纯静态页面，而是包含上传、轨迹解析、地点回填、天气聚合、景观预测、AI 对话、WebSocket、第三方 API 等复杂链路。
- 这类项目很多问题不会在编译时报错，而是会在“功能联动”阶段暴露。
- 因此，仅靠单元测试或本地手点几下并不够，需要一套更稳定的功能验证方式。

建议结论：

1. 先建立一份人工冒烟测试清单
2. 再把最核心的 3 条链路做成自动化冒烟测试
3. 然后按模块逐步补回归测试

## 2. 什么是冒烟测试

冒烟测试可以理解为：

“每次发版前，先确认系统最核心的功能还活着。”

它的特点是：

- 覆盖面广，但不要求测得很深
- 重点验证主功能能不能用
- 适合作为发版前、合并前、部署后的快速确认

对于 TrailQuest，冒烟测试最关注的不是细节，而是：

- 能不能登录
- 能不能发布路线
- 能不能打开路线详情
- 能不能发起 AI 对话
- 天气和景观模块能不能正常加载

## 3. 什么是回归测试

回归测试可以理解为：

“每次改动一个模块后，确认旧功能没有被顺手带坏。”

它的特点是：

- 更聚焦某次改动的影响范围
- 强调旧功能稳定性
- 适合在某个模块频繁迭代时持续补充

例如：

- 如果改了发布页地点回填，回归测试不应只看地点是否回填成功，还要检查发布是否成功、天气是否仍可查、AI 是否还能按地点推荐路线。
- 如果改了 AI 推荐地点检索，回归测试不应只看“萍乡”能否命中，还要检查“江西”“芦溪”“普通问答”是否都仍然正常。

## 4. 为什么 TrailQuest 特别需要这两类测试

### 4.1 项目里有很多跨链路模块

当前高风险链路包括：

- 发布路线：OSS + STS + 轨迹解析 + Geo 反查 + 路线创建
- AI 聊天：会话 + WebSocket + 模型 + 推荐结果 + 历史消息
- 天气/景观：Geo + Weather + Astronomy + 聚合降级
- 详情互动：点赞 + 收藏 + 评论 + 删除 + 登录态

这些链路有个共同点：

- 改一处，很容易影响另一处
- 问题往往不是“页面打不开”，而是“流程进行到一半失败”
- 这类问题最适合用冒烟测试和回归测试兜住

### 4.2 最近真实问题已经说明这件事

最近已经出现过多次典型的“需要冒烟/回归来兜底”的问题：

- 数据库 schema 升级不完整，导致发布路线失败
- AI 会话删除状态值和数据库枚举不匹配
- AI 流式收尾阶段误判失败
- 结构化地点检索导致“萍乡能查到、江西查不到”
- 景观预测依赖字段新增后，没有同步验证发布链路

这些问题都说明：

- 编译通过不等于功能稳定
- 单点修复不等于全链路无回归

## 5. TrailQuest 推荐的冒烟测试范围

建议先有一份固定的最小冒烟清单，每次发版前都走一遍。

### 5.1 P0 冒烟测试清单

#### A. 登录与基础进入

1. 用户可以正常登录
2. 登录后可以进入首页、搜索页、发布页、AI 聊天页
3. 登录态刷新后仍然有效

#### B. 发布路线主链路

1. 进入发布页
2. 上传封面图成功
3. 上传轨迹文件成功
4. 轨迹解析成功并自动回填地点
5. 点击发布成功
6. 发布后可进入路线详情页

#### C. 路线详情页主链路

1. 路线详情页可以正常打开
2. 天气模块正常返回
3. 景观预测模块正常返回
4. 点赞 / 收藏至少有一个操作成功
5. 评论发布成功

#### D. AI 聊天主链路

1. 进入 AI 聊天页
2. 新建会话成功
3. 发送普通问题成功
4. 发送推荐路线问题成功
5. 推荐路线卡片返回正常
6. 删除会话成功

### 5.2 冒烟测试执行建议

建议执行方式：

- 日常开发：人工快速走一遍
- 合并主分支前：至少走 P0 冒烟清单
- 线上/预发部署后：再走一遍冒烟确认

## 6. TrailQuest 推荐的回归测试范围

### 6.1 发布模块回归

当改动这些内容时，需要重点回归：

- 发布页布局
- 轨迹上传
- 轨迹解析
- 地点反查
- 结构化地点存储
- 路线创建 / 更新接口

推荐回归项：

1. 上传 GPX 是否成功
2. 上传 KML 是否成功
3. 地点是否自动回填
4. 用户手动修改 location 后是否仍能发布
5. 结构化地点字段是否仍正确落库
6. 发布后天气是否仍能按地点/坐标查询

### 6.2 AI 模块回归

当改动这些内容时，需要重点回归：

- WebSocket
- 会话列表
- 删除会话
- 推荐意图识别
- 地点检索
- 路线卡片返回

推荐回归项：

1. 普通问答能正常返回
2. “萍乡的路线有没有”能命中萍乡路线
3. “江西的路线有没有”能命中江西省内路线
4. “芦溪附近的路线”能命中区县级路线
5. 删除会话后列表正常刷新
6. 历史消息刷新不误判为整轮失败

### 6.3 天气与景观模块回归

当改动这些内容时，需要重点回归：

- Geo reverse / lookup
- 天气聚合
- 天文接口
- 景观预测输入特征

推荐回归项：

1. 有坐标的路线天气正常
2. 无坐标但有 location 的路线天气回退正常
3. 小时天气返回正常
4. 日出日落/天文数据缺失时不影响整体页面
5. 云海 / 雾凇 / 冰挂仍能正常展示

### 6.4 路线详情互动回归

当改动这些内容时，需要重点回归：

- 点赞
- 收藏
- 评论
- 删除评论
- 详情页状态同步

推荐回归项：

1. 点赞后计数更新
2. 收藏后状态更新
3. 评论发布成功
4. 删除自己的评论成功
5. 刷新页面后状态保持一致

## 7. 自动化建议：先轻后重

### 7.1 第一阶段：先做人工版

先做一份固定的人工清单，不一定立刻自动化。

优点：

- 成本最低
- 上手最快
- 能立刻减少“漏回归”

### 7.2 第二阶段：自动化冒烟先做 3 条

最值得优先自动化的 3 条链路：

1. 发布路线
2. AI 聊天
3. 路线详情互动

原因：

- 这三条最核心
- 也是最近出问题最多的地方
- 自动化后收益最大

### 7.3 第三阶段：模块化回归逐步补

后续谁改哪个模块，就顺手补这个模块的回归测试。

例如：

- 改发布页，就补发布回归
- 改 AI 推荐，就补 AI 回归
- 改天气聚合，就补天气/景观回归

这样成本可控，也更符合实际开发节奏。

## 8. 冒烟测试和回归测试与单元/集成/E2E 的关系

它们不是互斥关系，而是两个不同维度。

### 冒烟测试通常更像：

- 一小组高价值 E2E
- 少量关键接口联调
- 人工快速验收

### 回归测试通常更像：

- 单元测试
- 集成测试
- E2E 测试
- 人工回归清单

围绕某次改动范围做的“老功能确认”

所以更准确的理解是：

- 单元/集成/E2E 是测试形式
- 冒烟/回归 是测试策略和执行目的

## 9. 当前最推荐的落地顺序

如果只考虑投入产出比，建议按下面顺序推进：

### 第一优先级

1. 写一份 TrailQuest 冒烟测试清单
2. 固定每次合并前都走一遍
3. 补 3 条自动化冒烟：发布、AI、详情互动

### 第二优先级

1. 发布模块回归清单
2. AI 模块回归清单
3. 天气/景观模块回归清单

### 第三优先级

1. 把回归清单逐步沉淀为自动化用例
2. 配合埋点和错误监控做上线后验证

## 11. 最终建议

一句话总结：

TrailQuest 当前非常适合补“轻量但高价值”的冒烟测试和回归测试。

不需要一开始就做得很重，但必须尽快建立：

- 一份固定的人工冒烟清单
- 三条自动化核心冒烟链路
- 按模块维护的回归测试范围

这样你们后面继续高频迭代时，心里会稳很多，也更符合面试里被追问的“你如何证明功能可靠”的那套回答逻辑。
