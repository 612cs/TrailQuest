# TrailQuest 后台权限与治理设计文档

本文档用于定义 TrailQuest 后台系统下一阶段的权限边界、用户管理能力、路线审核能力、路线管理能力，以及“哪些允许软删除、哪些不应直接物理删除”的治理规则。

目标不是一次性做成复杂的中后台权限平台，而是给当前后台一期提供一套可直接开发、可逐步扩展、风险可控的治理方案。

## 1. 设计目标

本次后台权限设计聚焦四件事：

1. 明确管理员应该拥有的治理权力
2. 明确用户管理、路线审核、路线管理分别应支持哪些操作
3. 明确软删除、下架、封禁与物理删除的边界
4. 给后续接口、数据库字段和页面交互提供统一约束

本文档默认基于当前项目现状：

- 用户体系复用 `users` 表与现有 JWT 登录态
- 后台路径为 `apps/admin`
- 后端治理接口使用 `/api/admin/**`
- 路线当前已有审核能力：`review_status = pending | approved | rejected`
- 前台公开路线默认仅展示 `approved`

## 2. 当前后台角色模型

当前阶段只保留两类角色：

1. `USER`
2. `ADMIN`

第一阶段不做多级管理员权限，不引入超级管理员、审核员、运营、客服等更细的 RBAC 角色。

默认规则：

- `USER` 不能访问任何 `/api/admin/**`
- `ADMIN` 可以访问后台并执行治理动作
- 后续如要做多级权限，在不破坏当前接口的前提下扩展为 `permissions` 或后台角色表

## 3. 总体治理原则

后台治理动作分成四类：

1. 查看类：查看数据，不改变业务状态
2. 运营类：修改配置、审核内容、调整展示状态
3. 限制类：封禁、下架、隐藏、禁言
4. 删除类：分为软删除与物理删除

默认治理原则如下：

- 对“内容展示层”的处理优先使用软删除、下架、隐藏
- 对“用户账号”的处理优先使用封禁、停用，而不是物理删除
- 对“强关联主数据”原则上不允许普通管理员物理删除
- 所有具有破坏性的后台操作都应记录操作日志
- 所有治理动作都应支持明确的状态回退或恢复能力

## 4. 用户管理设计

### 4.1 用户管理页面应支持的操作

后台用户管理建议分为 `P0`、`P1`、`P2` 三层。

#### P0：立即需要支持

1. 查看用户列表
2. 关键词搜索用户
3. 按角色筛选用户
4. 查看用户基础资料
5. 查看用户发布路线数量
6. 查看用户注册时间
7. 查看用户当前账号状态
8. 封禁用户
9. 解封用户

#### P1：下一阶段建议支持

1. 查看用户详情页
2. 查看该用户发布的路线列表
3. 查看该用户评论列表
4. 查看该用户被举报记录
5. 批量隐藏该用户内容
6. 强制用户退出登录

#### P2：后续再考虑

1. 修改用户角色
2. 永久删除用户
3. 用户数据导出
4. 用户风险等级与人工备注

### 4.2 用户管理应新增的状态字段

当前 `users` 表只有 `role`，不够支持后台治理。建议补充：

- `status`：`active | banned | deleted`
- `ban_reason`
- `banned_by`
- `banned_at`
- `deleted_by`
- `deleted_at`

推荐含义：

- `active`：正常用户
- `banned`：封禁用户，禁止登录、禁止发布、禁止互动
- `deleted`：软删除用户，不再对前台作为活跃用户展示，但数据保留

### 4.3 用户管理行为规则

#### 封禁用户

管理员应有权封禁用户，但封禁不等于删除。

封禁后默认行为：

- 禁止登录
- 禁止发布路线
- 禁止评论、点赞、收藏等互动
- 历史内容默认不自动删除
- 可选是否同步隐藏其未审核或高风险内容

#### 解封用户

管理员应支持恢复用户状态：

- `banned -> active`
- 清空封禁时间与封禁管理员信息可选
- 保留封禁原因历史更合适，便于追溯

#### 删除用户

当前阶段不建议给管理员开放“物理删除用户”。

推荐策略：

- 普通 `ADMIN` 不允许物理删除用户
- 如确实要“删除用户”，第一阶段只做：
  - `status = deleted`
  - 账号失效
  - 前台不再作为正常用户展示
- 用户相关路线、评论、AI 会话、媒体文件仍保留

### 4.4 用户管理哪些属于软删除

属于软删除或限制类的操作：

- 封禁用户
- 停用用户
- 标记用户为已删除
- 隐藏用户主页
- 隐藏用户发布内容

### 4.5 用户管理哪些不应直接物理删除

以下内容当前阶段不应直接物理删除：

1. `users` 主记录
2. 用户发布的路线
3. 用户评论记录
4. 用户 AI 会话
5. 用户媒体文件记录
6. 用户审核与举报关联数据

原因：

- 这些数据跨表关联强
- 误删恢复成本高
- 治理和追责场景需要历史数据
- 物理删除容易造成路线、评论、会话“失去归属”

## 5. 路线审核设计

### 5.1 路线审核页面应支持的操作

路线审核应从“路线管理”中独立出来，作为单独的后台模块。

定位：

- 面向 `pending` 路线的内容审核工作台
- 聚焦“能不能上线”
- 不承担已上线路线的日常治理职责

后台路线审核建议按 `P0`、`P1`、`P2` 分层。

#### P0：立即需要支持

1. 查看待审核路线列表
2. 按关键词筛选
3. 按作者筛选
4. 查看路线详情
5. 审核通过
6. 驳回并填写原因

#### P1：下一阶段建议支持

1. 查看轨迹预览
2. 查看结构化地点
3. 查看天气摘要
4. 查看景观预测摘要
5. 查看审核历史
6. 批量审核

#### P2：后续再考虑

1. 审核分配
2. 多审核员协作
3. 审核 SLA 与统计报表

### 5.2 路线审核列表交互建议

路线审核列表的默认交互应改为：

- 点击整条路线记录进入详情页
- 不再额外放一个“查看详情”按钮作为主入口

推荐原因：

- 更符合审核工作流，减少多一步点击
- 列表作为审核工作台，操作密度更高
- 后续如果需要补充快捷操作，可以在行尾保留“通过 / 驳回”次入口，但主入口仍应是点击整行

推荐列表字段：

- 封面缩略图
- 路线名
- 作者
- 位置
- 发布时间
- 当前审核状态

### 5.3 路线审核行为规则

#### 审核通过

管理员可执行：

- `review_status = approved`
- 记录 `reviewed_by`
- 记录 `reviewed_at`
- 前台公开列表即可展示

#### 驳回路线

管理员可执行：

- `review_status = rejected`
- 写入 `review_remark`
- 记录审核人和时间
- 前台公开列表不展示
- 作者可在编辑后重新提交审核

### 5.4 路线管理设计

路线管理与路线审核应分成两个后台模块。

定位：

- 面向已经进入系统的路线内容治理
- 聚焦“上线后的管理、下架、恢复、运营处理”
- 不负责待审核路线的首轮审批

### 5.5 路线管理页面应支持的操作

后台路线管理建议按 `P0`、`P1`、`P2` 分层。

#### P0：立即需要支持

1. 查看已上线和已下架路线列表
2. 按状态筛选
3. 按关键词筛选
4. 按作者筛选
5. 点击路线进入详情页
6. 下架路线
7. 恢复路线

#### P1：下一阶段建议支持

1. 查看轨迹预览
2. 查看结构化地点
3. 查看天气摘要
4. 查看景观预测摘要
5. 标记推荐路线或精选路线
6. 标记违规路线
7. 查看操作历史

#### P2：后续再考虑

1. 批量下架
2. 物理删除路线
3. 数据修复工具

### 5.6 路线状态模型建议

当前路线已有：

- `status`：`active | deleted`
- `review_status`：`pending | approved | rejected`

为了支持后台治理，建议沿用双状态设计：

#### 展示/存在状态

- `status = active`：记录存在
- `status = deleted`：软删除，不在前台公开展示

#### 审核状态

- `pending`：待审核
- `approved`：审核通过
- `rejected`：审核驳回

后续如需要更细，可以扩展审核状态为：

- `pending`
- `approved`
- `rejected`
- `offline`

但第一阶段为了兼容现有实现，不建议立刻改 `review_status` 枚举。更稳妥的是：

- 继续用 `approved` 表示可展示
- 使用 `status = deleted` 实现下架/隐藏

### 5.7 路线管理行为规则

#### 审核通过

管理员可执行：

- `review_status = approved`
- 记录 `reviewed_by`
- 记录 `reviewed_at`
- 前台公开列表即可展示

#### 驳回路线

管理员可执行：

- `review_status = rejected`
- 写入 `review_remark`
- 记录审核人和时间
- 前台公开列表不展示
- 作者可在编辑后重新提交审核

#### 下架路线

管理员应支持“下架”，但当前阶段建议实现为软删除：

- `status = deleted`
- 不影响路线原始记录、轨迹、评论、媒体文件
- 后台仍可查看和恢复
- 前台公开列表与搜索不再展示

#### 恢复路线

管理员应支持恢复：

- `status = active`
- 若 `review_status = approved`，恢复后前台重新可见
- 若 `review_status != approved`，恢复后仍不公开展示

### 5.8 路线管理哪些属于软删除

属于软删除或隐藏类的操作：

- 下架路线
- 隐藏路线
- 驳回路线
- 从公开列表移除路线

推荐落地方式：

- 使用 `status = deleted` 实现“下架/隐藏”
- 使用 `review_status = rejected` 实现“审核不通过”

### 5.9 路线管理哪些不应直接物理删除

以下内容当前阶段不应允许普通管理员物理删除：

1. `trails` 主记录
2. `trail_tracks` 轨迹解析记录
3. `media_files` 封面、图库、轨迹媒体记录
4. 路线评论与互动数据
5. 审核记录与审核备注
6. 景观预测与天气相关快照

原因：

- 路线是强业务主数据
- 和轨迹、图片、评论、收藏、点赞、AI 推荐等多处关联
- 直接物理删除容易造成数据孤儿与统计失真
- 后续治理、复审、追责都需要保留原始记录

## 6. 删除策略总表

### 6.1 可以先做成软删除的

1. 用户账号停用
2. 用户封禁
3. 用户页面隐藏
4. 路线下架
5. 路线隐藏
6. 评论删除
7. 举报处理后的前台隐藏

### 6.2 当前阶段绝不建议直接物理删除的

1. 用户主记录
2. 路线主记录
3. 轨迹记录
4. 媒体文件主记录
5. AI 会话与消息
6. 审核记录
7. 举报记录

### 6.3 可以考虑物理删除但需严格控制的

仅限以下情况：

1. 测试数据
2. 无关联脏数据
3. 重复导入失败的数据
4. 后台明确标记为“可清理”的垃圾文件记录

并且必须满足：

- 只有更高权限角色可操作
- 二次确认
- 强提醒不可恢复
- 操作日志落库

## 7. 建议新增的后台接口

### 7.1 用户管理接口

建议新增：

- `GET /api/admin/users`
- `GET /api/admin/users/{id}`
- `POST /api/admin/users/{id}/ban`
- `POST /api/admin/users/{id}/unban`
- `POST /api/admin/users/{id}/deactivate`

后续可扩：

- `GET /api/admin/users/{id}/trails`
- `GET /api/admin/users/{id}/reviews`
- `POST /api/admin/users/{id}/hide-content`

### 7.2 路线审核接口

当前已有：

- `GET /api/admin/trails`
- `GET /api/admin/trails/{id}`
- `POST /api/admin/trails/{id}/approve`
- `POST /api/admin/trails/{id}/reject`

建议后续按模块语义拆分为：

- `GET /api/admin/trail-reviews`
- `GET /api/admin/trail-reviews/{id}`
- `POST /api/admin/trail-reviews/{id}/approve`
- `POST /api/admin/trail-reviews/{id}/reject`

可选后续：

- `POST /api/admin/trail-reviews/batch-approve`

### 7.3 路线管理接口

建议新增：

- `GET /api/admin/trail-management`
- `GET /api/admin/trail-management/{id}`
- `POST /api/admin/trail-management/{id}/offline`
- `POST /api/admin/trail-management/{id}/restore`

可选后续：

- `POST /api/admin/trail-management/batch-offline`

## 8. 页面交互建议

### 8.1 用户管理页

建议操作区包含：

- 查看详情
- 封禁用户
- 解封用户
- 停用账号
- 查看其发布路线

交互要求：

- 封禁、停用需二次确认
- 必须填写或选择原因
- 操作成功后弹窗或 toast 提示
- 列表即时更新状态

### 8.2 路线审核页

建议操作区包含：

- 审核通过
- 驳回

交互要求：

- 点击整条路线记录进入详情页
- 不再依赖单独的“查看详情”按钮
- 驳回必须填写审核备注
- 审核结果应有清晰成功提示

### 8.3 路线管理页

建议操作区包含：

- 下架
- 恢复
- 查看详情

交互要求：

- 点击整条路线记录进入详情页
- 不再依赖单独的“查看详情”按钮
- 下架需二次确认
- 恢复时需明确提示“是否恢复公开展示”
- 下架与恢复结果应有清晰成功提示

## 9. 审计日志建议

凡是以下后台动作，建议都进入操作日志：

1. 审核通过路线
2. 驳回路线
3. 下架路线
4. 恢复路线
5. 封禁用户
6. 解封用户
7. 停用用户
8. 删除评论
9. 处理举报
10. 修改首页大图等站点设置

日志至少记录：

- 操作类型
- 操作对象类型
- 操作对象 ID
- 操作人 ID
- 操作备注
- 操作时间

## 10. 推荐开发优先级

### P0：优先开发

1. 路线审核模块与路线管理模块拆分
2. 路线下架 / 恢复
3. 用户封禁 / 解封
4. 用户状态字段
5. 后台操作日志骨架

### P1：接着开发

1. 用户详情页
2. 用户发布路线列表
3. 用户评论查看
4. 路线操作历史

### P2：后续再做

1. 更细粒度后台权限
2. 批量治理能力
3. 物理删除高权限控制
4. 风险账号与内容联动规则

## 11. 最终结论

对于 TrailQuest 当前阶段，管理员应该拥有对用户和路线的治理权，但这种权力应优先设计为“可恢复、可追溯、低误伤”的形式。

推荐落地原则是：

- 用户：先做封禁、停用、隐藏，不做物理删除
- 路线审核：单独模块，负责待审核路线审批
- 路线管理：单独模块，负责已上线路线的下架、恢复与治理
- 路线：整体上不做物理删除
- 评论：允许后台删除，但建议保留审计记录
- 举报：允许后台处理，但处理结果应可回溯

一句话总结：

- 管理员应该有治理权
- 但当前阶段不应该有大范围物理删除权
- 优先使用软删除、下架、封禁、隐藏来实现后台管理能力
