ABSTRACT

As outdoor activities such as hiking, mountaineering and camping gain popularity, users increasingly need better tools for route discovery, weather assessment and risk evaluation before trips. However, existing map services focus on urban road navigation while tourism platforms center on ticketing and hotel booking — neither adequately covers route geometry, elevation profiles, gear requirements or dynamic weather conditions specific to outdoor trails. As a result, outdoor enthusiasts often resort to piecing together fragmented information from social media posts, track files and user comments, which is inefficient and introduces uncertainty into route planning.

To address this gap, this thesis designs and implements an intelligent outdoor route recommendation platform based on Vue. The system adopts a separated frontend-backend architecture: the frontend is built with Vue 3, TypeScript, Tailwind CSS and DaisyUI; the backend uses Spring Boot 3, MyBatis-Plus, MySQL, Redis and Spring Security for business logic, data management and access control. For intelligent services, the platform integrates the DeepSeek large language model with Function Calling, enabling users to search for routes through natural language conversation — the model interprets user intent and extracts structured parameters, while the backend handles actual route retrieval and ranking. Additionally, the system leverages the Amap API for track visualization and geolocation, and combines weather data with a random forest model to estimate the occurrence probability of four types of natural scenery: sea of clouds, sunrise/sunset, rime ice and ice hanging.

Testing results show that the platform delivers core functions including route browsing with multi-condition filtering, route publishing with GPX/KML track parsing, community commenting and interaction, AI-powered conversational recommendation, weather and landscape prediction, and admin management. All major business workflows operate as intended, meeting the functional verification requirements of an undergraduate capstone project.
**KEYWORDS**: Outdoor Route Recommendation; Vue 3; Spring Boot 3; DeepSeek Large Language Model; Landscape Prediction

1.1 研究背景

近几年，登山、徒步、露营、越野跑等户外运动在国内发展很快，参与人群从专业玩家扩展到了普通大众。国家也陆续出台了多项政策，推动体育和旅游产业融合，鼓励用云计算、大数据、人工智能这些技术来改善旅游公共服务。用户出门前不光想知道"路在哪"，还想了解"这条路难不难""明天天气适不适合""有没有机会看到云海日出"，对路线信息的完整度和个性化程度都有了更高的期待。

但现有的数字化工具在户外场景下其实还有不少短板。传统的旅游电商平台主要围绕景区门票、酒店预订这类服务设计，数据模型偏固定，很难把徒步路线里的轨迹走向、累计爬升、路面类型、装备建议、天气窗口这些要素完整表达出来。常用的地图导航虽然擅长城市路网的路径规划，一旦走进山路、峡谷、森林步道，对高程变化、非铺装路面、局部小气候这些因素的考虑就很有限。很多户外爱好者还是得靠自己翻社交平台上的游记、轨迹文件、评论区来拼凑信息，做路线选择和风险评估的时候不确定因素比较多。

基于这样的实际情况，做一个专门面向户外路线的智能推荐平台，把路线检索、轨迹展示、社区分享、AI 对话和景观预测整合到一起，既有实际应用场景，工程实现上也有一定挑战。本文尝试补上当前户外出行服务在信息整合和智能推荐方面的缺口。
```

1.2 研究意义

本课题的工作主要集中在两个层面。

技术层面，本文尝试了 AI 和 GIS 在户外路线推荐场景下的一种协作方式。大语言模型擅长理解自然语言、处理模糊表达和多轮对话，但直接用它做精确的地理空间计算或者时序气象分析并不合适，而且每条检索都走大模型 API 的话，响应延迟和调用费用都是实际问题。所以本文采用的是一种混合方案：能用数据库查询和后台规则直接处理的标准化筛选，就不走大模型；用户输入的模糊、复杂需求，由大模型先解析成可执行的参数，再由后台去查路线库、做排序和组装结果。这样做的好处是语义交互的能力保留住了，系统的响应速度和成本也更可控。

应用层面，平台想把对话推荐、路线画像和环境信息打通，让推荐结果不只是冷冰冰的列表，而更像一个了解用户偏好的助手给的建议。系统会参考用户的户外经验、负重习惯、路线难度、距离、爬升和天气条件来筛选候选路线，返回的时候附上简单的推荐理由。这种从"用户自己翻找"到"系统帮助筛选"的转变，能帮用户省下不少在碎片信息中来回比对的时间。另外天气和景观预测模块也可以在出行前提示降雨、低能见度等风险，让用户在做行程判断的时候多一个参考维度。
```

1.3.1 国外研究现状

国外在路线规划和旅游推荐方面的研究起步比较早，已经从经典的最短路径算法扩展到考虑场景质量、用户偏好、动态约束这些因素的综合推荐。Siriaraya 等人对步行导航做过一篇质量感知路线的综述，指出景观、安全、舒适、可达性和环境体验已经是路线推荐中比较重要的评价维度[1]。在旅游推荐方面，Zhang 从大数据角度讨论了智能旅游路线的推荐方法，Liu 和 Wang 分别尝试把知识图谱引入到个性化旅游路线推荐中[2-4]。后来的研究更加重视多源数据和算法模型的融合，比如 Su 等人提出了基于知识图谱的中文旅游个性化推荐算法，Zheng 等人从多目标演化算法的角度做旅游路线推荐，在个性化生成和多约束排序方面提供了新的思路[19-20]。

大语言模型兴起之后，国外开始尝试把自然语言交互和工具调用引入推荐系统。Wang 等人的 ToolFlow 工作通过合成自然连贯的工具调用对话，提升了模型在多步骤任务中的表现[8]；Qin 等人则研究了通用大模型在开放域场景下 Function Calling 能力的释放方式[9]。从这些研究来看，大模型不仅可以生成文本，还能作为意图理解和外部服务编排的枢纽，这为路线推荐从"关键词搜索"升级成"自然语言理解+工具链执行"提供了方法上的参考。

在推荐接受机制方面，Li 等人从 Quantified Self 的角度研究了消费者对 AI 推荐的接受过程，发现用户画像、自我量化数据和推荐解释会影响用户对推荐结果的信任程度[17]。这个研究对本文的启发是：做智能推荐不能只关心结果有没有命中条件，还得给出推荐理由、环境风险说明和追问建议。户外路线推荐场景下，用户经常要在时间、体力、天气、景观和安全之间做权衡，所以推荐结果需要呈现得更直观、更容易比较、支持继续追问。

在环境预测方面，Kim 等人用随机森林对韩国区域的能见度做了预测，证明了机器学习方法在低能见度场景预测中是可行的[10]；Lakra 和 Avishek 对雾的形成因素、分类、预测和检测做了系统综述，梳理了湿度、温度、风速、气溶胶等因素对雾和低能见度天气的影响[11]；Parde 等人从集合预报的角度构建了雾概率预报的决策支持系统[21]。这些工作表明，把气象数据、地理信息和机器学习模型结合起来做云海、雾凇等户外景观的概率预测，在技术上是走得通的。

总的来说，国外研究在路线推荐、大模型工具调用和环境预测方面已经积累了不少成果，但这些工作分散在不同的研究领域。直接面向中文户外徒步场景，同时把 GIS 地图、天气数据、用户画像、AI 对话和景观预测集成到一套系统里的研究，目前还不多见。
```

### 2.1 前端技术栈（替换原第159-163行）

2.1 前端技术栈

本平台前端选用 Vue 3 作为主要框架，配合 TypeScript 和 Tailwind CSS 完成开发。

选择 Vue 3 主要考虑了它的 Composition API。相比传统的 Options API，Composition API 可以按功能逻辑来组织代码，而不是按组件选项拆分。像路线筛选、地图交互、文件上传、AI 对话这些功能模块，用 Composition API 封装后更容易复用和维护。Vue 3 底层用 Proxy 做响应式，和 Vue 2 的 Object.defineProperty 方案相比，对新增属性、数组操作和深层嵌套对象的响应式处理也更完善。对于本系统里涉及轨迹坐标数组、筛选条件对象和对话状态这类复杂数据结构，Proxy 方案确实省了不少事。

TypeScript 的引入主要是为了解决 JavaScript 动态类型在实际开发中容易出的问题。给登录信息、路线详情、评论列表、上传结果、AI 消息这些接口都定义了类型之后，编译阶段就能拦住一部分字段缺失和类型不匹配的 bug，前后端对接的时候也少踩一些坑。

UI 层面用了 Tailwind CSS 配合 DaisyUI 组件库。Tailwind 的原子化 class 在路线卡片、筛选面板、图片列表、后台表格这些地方用起来比较顺手，间距、颜色和响应式适配都能保持一致。Vite 作为构建工具，冷启动和热更新都很快，项目里前端代码放在 apps/web 目录下，开发、构建和测试命令都围绕这个目录配置好了。
```

### 2.2 后端框架与接口设计（替换原第164-167行）

2.2 后端框架与接口设计

后端方面，系统基于 Spring Boot 3 来构建 RESTful API。Spring Boot 3 跑在 Java 17 和 Jakarta EE 规范上，Web 接口、依赖注入、配置管理、安全认证这些后端开发中的常见需求都有比较成熟的组件支持。项目的后端代码放在 apps/api 目录里，按 Controller、Service、Mapper 和 Config 分层组织，分别处理用户认证、路线管理、评论互动、媒体上传和后台审核这些业务。

接口规范上，本系统按 RESTful 风格来设计。用户、路线、评论、天气和景观预测这些业务资源都通过统一的 HTTP 接口暴露，用 GET 做查询、POST 做创建、PUT 做更新、DELETE 做删除。无状态的接口设计对前后端协作、接口测试、权限控制和后续扩展都比较友好。

Redis 在目前的实现里主要用于 OSS 上传凭证的短期校验。后端签发 OSS STS 临时凭证的时候，把 objectKey、用户 ID 和业务类型写入 Redis，设一个和 STS 凭证同步的过期时间。前端回传上传结果时，后端先从 Redis 里查 objectKey 记录，校验用户和业务类型是否匹配，文件地址是否合法。媒体文件写入数据库成功后删掉对应的 Redis 键，这样同一个上传凭证就不能被重复利用。这个设计不是用 Redis 替代 MySQL 做持久化，只是在上传这个环节加了一道短期状态校验。如果后面要做热门路线缓存、首页配置缓存之类的性能优化，Redis 的扩展空间是现成的。
```

```
2.3 人工智能与 GIS 技术

本系统的智能部分主要依赖三块：大语言模型、地理信息服务、景观预测模型。

智能交互方面，系统接入了 DeepSeek 大语言模型来处理用户的自然语言输入。用户可以用日常说话的方式描述想去的地方、出行时间、能接受的难度、想看的景观类型，模型负责从这些自然语言中抽取出地点、时间、难度、偏好等关键信息。和传统的关键词搜索比起来，大模型在处理模糊表达、条件组合、口语化问法这些场景时的表现要好不少。最近关于 ToolFlow 和 Meta-Tool 的研究也说明，LLM 在结构化任务编排和外部工具协同方面有不小的拓展空间[8-9]。

为了让大模型的语义理解和后台的业务能力真正联动起来，系统引入了 Function Calling。简单说就是：模型理解用户意图后，按预定义的 Schema 把解析出来的参数组织成结构化的调用请求，后台收到请求后去查路线库、做筛选和排序，最终生成推荐结果。这样整个链路就闭环了——自然语言进来，推荐卡片出去，中间环节都是确定性的。

GIS 方面，系统接入高德地图 API 来完成地图渲染、位置解析和路线轨迹可视化。路线详情页里的起终点标注、轨迹线绘制、位置提示这些都靠高德地图的能力。前端已经搭好了地图相关的页面组件，后端也围绕地理和天气数据组织了一批辅助接口。

景观预测这块，后台用 Spring Boot 服务集成了随机森林预测器，再加上一些基于经验的规则修正逻辑。输入是时间、天气、地形等数据，输出是云海、雾凇、冰挂、日出日落这些景观的出现概率和风险等级。这让平台在做路线推荐的同时，也能给用户提供一些环境辅助判断信息。

总的来说，前端 Vue 3 技术栈负责交互和展示，Spring Boot 后端支撑业务和安全，AI 和 GIS 构成智能服务的核心。
```

```
4.4 AI 大模型与算法集成设计

自然语言到路线检索的匹配机制

当用户在 AI 助手里输入一段自然语言后，系统先走一道本地判断：如果请求里带了明确的地名，而且条件基本是标准化的（比如"长沙岳麓山难度初级路线"），就直接走数据库索引和字段匹配，不调用大模型。只有当请求比较模糊、条件组合比较复杂（比如"周末带小孩去爬山，不要太累，最好能看到好看的风景"），才把请求转给大模型做语义解析。

大模型收到请求后，根据预设的 System Prompt 和 Function Calling Schema 来完成语义抽取和参数映射，输出结构化的查询参数。后台再用这些参数去查路线库、做排序和封装结果。这种分层处理既保留了大模型处理复杂意图的能力，也避免了所有请求都走大模型带来的延迟和费用[1-4][8-9]。

景观概率预测的数据处理流程

景观预测模块在后端集成随机森林预测器，并结合规则修正输出云海、雾凇、冰挂和日出日落的概率和等级。随机森林是一种集成学习算法，通过多棵决策树投票来做分类和概率估计，适合处理气温、湿度、风速、云量、海拔等多因素交叉影响的环境预测问题。Kim 等人之前就用随机森林做过韩国的能见度预测，说明这个方法在气象相关预测上是可行的[10]。

数据处理链路是：先根据路线起点坐标或用户输入的文本信息解析出经纬度，然后拉取和风天气的实况和逐小时预报数据、天文台的日月出落时刻以及光污染地图数据。接着做特征工程——把湿度、云量、风速、露点差、海拔和降水等变量编码成模型输入。Lakra 和 Avishek 在综述里指出，湿度、边界层结构、地形和凝结核条件对雾的形成和能见度变化很关键[11]，所以在选特征的时候也参考了这些结论。模型输出概率之后，再经过一层基于气象常识的规则修正（比如降水量超过某个阈值直接降低云海概率），最终返回给前端展示。
```

### 7.2 存在的问题与未来展望（替换原第1706-1709行）

```
7.2 存在的问题与未来展望

系统虽然能跑起来，但距离一个真正好用的产品还有不少差距，主要在两个方面：

第一是路线数据的丰富度。现在路线推荐主要靠里程、爬升、难度、天气和用户反馈这几个维度，对路线本身的"质感"——比如路面是土路还是碎石路、沿途植被怎么样、有没有危险路段——这些信息还没有。以后如果能积累足够用户授权的图片和视频数据，可以用图像识别的方法从用户上传的媒体内容里自动提取地面类型、路况和风险特征，把这些补充到路线画像里，推荐的时候就有更多维度可以参考了。

第二是景观预测的准确性。现在的随机森林模型只是完成了基本验证，输入特征、训练数据和预测准确性都还有很大的提升空间。后面如果能积累更长时间段的天气和实地观测数据，可以试试时序模型或者引入地形空间特征——比如同一个地区不同海拔的微气候差异。但前提是数据要先够用，数据不够的时候上复杂模型意义不大。

系统代码方面也有一些要改进的地方，比如部分模块的错误处理不够细致，前端有些页面的加载状态处理比较粗糙。这些不属于架构层面的问题，后续迭代逐步修复即可。
