# 4 系统总体设计

## 4.1 系统架构设计

### 4.1.1 前端展示层架构

结合任务书、开题报告和现有论文的整体思路，本系统采用前后端分离的总体架构，并在前端展示层中承担页面呈现、交互控制、状态同步和地图可视化等职责。

用户侧前端主要围绕首页、搜索页、社区页、路线详情页、发布页、AI 助手页、路线图片展示页和个人中心页展开，通过组件化方式完成路线卡片、筛选面板、评论列表、消息流、上传表单和地图容器等可复用界面组织。

后台前端则作为独立管理端存在，负责后台首页、路线审核、路线管理、用户治理、评论治理、举报处理、设置中心、配置中心和操作日志等页面的展示与交互。

在前端状态管理上，系统通过路由与状态存储相结合的方式组织页面数据流，使筛选条件、登录态、AI 会话状态、用户画像和后台权限信息能够在多页面场景下保持一致。

在地图展示方面，前端通过 GIS 可视化组件承接路线轨迹、起终点、地点信息和天气景观提示的页面表达，使用户能够在查看图文内容的同时完成空间感知。

在媒体上传方面，前端发布页和个人资料页负责与上传服务协同，完成封面图、路线相册、评论图片和轨迹文件的上传、预览与提交流程。

### 4.1.2 后端服务层架构

后端服务层采用 Spring Boot 分层组织方式，围绕控制器层、服务层、数据访问层和统一响应层完成业务处理。

控制器层负责接收前端请求并进行接口路由分发，主要包括用户认证控制器、路线控制器、评论控制器、上传控制器、AI 控制器、管理员控制器和后台配置控制器。

服务层负责承接核心业务逻辑，包括用户资料维护、路线发布与审核、轨迹解析、天气查询、景观预测、AI 会话管理、配置维护和治理日志记录等。

数据访问层负责与 MySQL 进行交互，完成用户、路线、评论、媒体、配置、景观特征和 AI 会话等结构化数据的持久化处理。

在认证与权限控制方面，系统通过 Spring Security 与 JWT 机制实现用户态与管理员态隔离，并通过角色校验保障后台治理接口仅能由管理员访问。

在缓存与性能支持方面，Redis 作为辅助存储介入热点路线、首页配置、会话上下文、上传断点和后台统计等高频访问场景，以降低数据库压力并提升整体响应效率。

### 4.1.3 AI 服务层架构

AI 服务层是本系统总体设计中的亮点部分，也是任务书和开题报告中明确要求重点实现的能力层。

按照既有材料确定的技术路线，系统在 AI 服务层中保留 DeepSeek 大模型、Function Calling 机制和 Python Flask 随机森林预测服务三个核心组成部分。

其中，DeepSeek 负责对用户自然语言进行语义理解和意图识别，Function Calling 负责将模型输出转化为内部可执行的路线检索与环境查询能力调用，Python Flask 服务则负责承接天气与景观相关的随机森林预测推理。

从当前工程实现看，仓库已经具备 AI 会话管理、流式消息返回、路线卡片嵌入、天气服务、景观特征构造和预测结果封装等支撑基础，这些内容能够作为既有技术路线的工程补强。

因此，AI 服务层在总体架构中的定位，不是孤立的大模型调用，而是围绕“语义理解、业务检索、GIS 联动、天气支撑、景观预测和前端结果展示”形成完整闭环。

## 4.2 系统功能设计

### 4.2.1 前台用户功能设计

前台用户功能围绕“找路线、看路线、发路线、聊路线、管个人信息”五类核心行为展开。

在账号与个人信息方面，系统支持用户注册、登录、个人资料维护和徒步画像配置，为个性化推荐和后续交互提供基础数据。

在路线浏览方面，首页负责展示热门路线、主题入口和推荐内容，搜索页负责根据关键字、难度、距离、时长和负重偏好等条件筛选路线，路线详情页则集中展示图文内容、轨迹地图、天气信息、景观预测和评论互动。

在互动能力方面，用户可以对路线进行点赞、收藏、评论、回复和图片浏览，也可以在社区页查看其他用户发布的路线体验和反馈内容。

在内容生产方面，用户可以在发布页填写路线基础信息、上传封面与相册、提交轨迹文件并生成完整路线内容。

在智能交互方面，用户可以进入 AI 助手页，通过自然语言直接描述需求，由系统输出路线推荐结果、追问建议和路线卡片，从而实现比传统筛选更灵活的检索方式。

### 4.2.2 后台管理员功能设计

后台管理员功能围绕“审核、治理、配置、审计”四条主线展开。

在后台首页中，系统需要为管理员展示路线审核数量、用户状态统计、治理风险提示和待处理事项，以便管理员快速掌握平台运行情况。

在路线审核与路线管理方面，管理员需要能够查看待审核路线列表、进入审核详情页完成通过或驳回操作，也能够对已上架路线执行下架、恢复和详情复核等管理动作。

在用户治理方面，管理员应能够查看用户列表、核验账号状态、执行封禁与解封操作，并对异常账号进行风险控制。

在评论与举报治理方面，管理员应能够查看评论内容、处理隐藏与恢复、执行删除或批量治理，同时对举报信息进行核查与处置。

在系统运营方面，管理员还需要维护首页大屏设置、站点运营配置、系统选项组和选项项，以支撑前台页面的动态化配置和持续维护。

在审计追踪方面，系统通过操作日志记录管理员动作、对象、原因、结果和快照，为平台治理过程提供可追溯依据。

### 4.2.3 大模型与 GIS 功能设计

大模型与 GIS 功能设计是本系统与普通路线展示平台相比最具有特色的部分。

大模型部分主要负责理解用户自然语言中的地点、时间、难度、轻重装偏好、景观诉求和出行目标，并将这些信息组织为结构化参数。

GIS 部分主要负责路线地图展示、地理位置解析、坐标上下文维护和目标区域空间可视化，使路线检索结果不再停留在文本描述层面，而能够转化为更具空间感知能力的服务形式。

在此基础上，天气与景观联动功能将地理位置、天气数据、天文信息和景观预测能力融入路线详情与智能推荐过程，使平台既能够“推荐去哪儿”，也能够“解释什么时候更适合去、可能看到什么”。

综合来看，大模型与 GIS 功能并非附加模块，而是贯穿路线检索、详情浏览、AI 对话和环境辅助决策的核心能力层。

## 4.3 数据库设计

### 4.3.1 概念模型设计

从概念层面看，系统中的核心实体包括用户、徒步画像、路线、路线图片、路线轨迹、标签、评论、媒体文件、上传断点、AI 会话、AI 消息、系统配置、后台日志以及景观特征与观测标签等对象。

用户与徒步画像之间构成一对一关系，用于描述用户在户外活动中的经验和偏好。

用户与路线之间构成发布关系，用户与评论之间构成发表关系，用户与点赞收藏记录之间构成交互关系。

路线与路线图片、路线轨迹、评论、标签、景观特征快照和景观观测标签之间均构成一对多或多对多关系，这些关系共同构成路线详情页面与推荐服务所依赖的数据骨架。

AI 会话与 AI 消息之间构成一对多关系，用于支撑聊天模块中的多轮对话能力。

后台配置实体与首页运营、搜索筛选、用户画像选项等页面行为相关联，后台操作日志则与各类治理动作形成审计关系。

### 4.3.4 Redis 缓存数据结构设计

在缓存设计方面，Redis 主要服务于热点数据快速读取、短周期上下文维护和上传过程状态保持。

从论文整体设计角度看，Redis 不仅用于提高接口响应速度，也有助于缓解数据库在高频查询场景下的压力。

考虑到毕业设计写作需要展示完整的缓存思路，本文对 Redis 数据结构采用模拟设计方式进行说明，其核心缓存对象如下表所示。

| Key 模式 | 数据结构 | 示例内容 | 过期策略 | 设计用途 |
| --- | --- | --- | --- | --- |
| `home:hot_trails:{city}:{page}` | `STRING(JSON)` | 热门路线卡片列表 | `10~30` 分钟 | 缓存首页热门路线，减少高频分页查询 |
| `home:hero_config` | `HASH` | 首页大图地址、标题、副标题 | `30` 分钟 | 缓存首页运营配置，降低配置表读取频次 |
| `ai:conversation:context:{userId}:{conversationId}` | `HASH` | 当前会话上下文、最近意图、追问状态 | `30` 分钟滑动过期 | 保存 AI 对话临时上下文，减少重复拼接历史消息 |
| `upload:checkpoint:{userId}:{bizType}:{fileName}` | `HASH` | `uploadId`、已完成分片、对象键 | 至少到 `expires_at` | 支撑大文件断点续传和异常恢复 |
| `trail:interaction:{trailId}` | `HASH` | 点赞数、收藏数、评论数 | `5~10` 分钟 | 缓存路线互动统计，提升详情页加载速度 |
| `admin:dashboard:summary` | `HASH` | 审核数、封禁用户数、待处理事项 | `5` 分钟 | 提升后台首页统计接口响应速度 |

需要说明的是，Redis 中的数据以缓存和辅助状态为主，MySQL 仍然是系统的权威数据源。

因此，缓存设计遵循“可失效、可重建、以数据库最终一致性为准”的原则，以避免因缓存失真影响核心业务正确性。

## 4.4 AI 大模型与算法集成设计

### 4.4.1 大模型语义理解与意图识别设计

在智能推荐链路中，系统首先接收用户的自然语言输入，再由大语言模型对用户意图进行识别与结构化表达。

按照既有技术路线，系统以 DeepSeek 大模型作为语义理解核心，重点识别地点约束、难度偏好、负重倾向、出行时长、风景诉求和用户想要完成的具体任务。

为了提高语义识别的稳定性，系统在提示词中同时引入用户画像摘要、历史会话记录和当前页面上下文，使模型不只是理解一条孤立消息，而是在“用户是谁、此前聊过什么、当前页面处于何种场景”的综合条件下进行判断。

在输出形式上，系统要求模型优先生成结构化结果，而不是直接返回长文本，从而便于后续 Function Calling 或内部检索链路继续处理。

这种设计可以避免大模型直接生成不受约束的虚构路线信息，也有助于将自然语言交互稳定映射到平台内部真实可查询的数据资源上。

### 4.4.2 Function Calling 与路线检索协同设计

完成意图识别后，系统通过 Function Calling 机制将模型输出转换为内部可执行的业务调用。

在论文设计口径上，该机制可以抽象为 `searchTrails`、`getTrailWeather`、`getTrailPrediction` 和 `getUserPreference` 等函数能力集合，大模型只负责决定“该调什么、带什么参数”，真正的数据查询与结果组装仍由业务系统完成。

在当前工程实现中，系统已经具备结构化参数解析、路线检索参数封装、候选路线排序和路线卡片生成等基础能力。

用户消息经过解析后，会被转化为地点、难度、时长、距离、标签和关键字等参数，再交由路线服务查询候选路线。

若精确地理匹配结果不足，系统还会执行位置文本回退检索，以提高召回率。

在排序阶段，系统结合地理匹配程度、难度一致性、装备匹配程度、用户画像偏好、路线评分、点赞数和收藏数等因素进行综合打分，从而获得更符合用户语义诉求的推荐结果。

最终生成的结果不是单纯文本，而是带有路线名称、地点、难度、时长、距离、收藏点赞状态和推荐理由的路线卡片，这些卡片会被嵌入对话流中返回前端。

### 4.4.3 GIS 地理信息与天气服务协同设计

GIS 能力在本系统中承担着路线地图表达、地理位置解析和天气景观联动三个方面的职责。

在路线地图展示层面，系统通过高德地图 API 将路线起终点、轨迹线、位置范围和地理环境提示可视化，使用户能够从空间维度理解路线。

在位置解析层面，系统优先使用路线轨迹中记录的起点经纬度作为天气与景观分析的定位依据。

当路线缺少明确坐标时，系统再根据路线文本地点进行地理解析，以保证天气服务和环境服务具备尽可能稳定的定位来源。

在天气服务协同层面，系统将 GIS 位置结果传递给天气服务、小时天气服务、天文服务和光污染服务，进一步得到七日天气、小时天气、日出日落、月相和夜空环境等信息。

这些结果一方面在路线详情页直接展示，另一方面也会作为景观预测和 AI 推荐解释的重要环境输入，从而使平台在路线浏览、环境判断和智能解释之间形成统一链路。

### 4.4.4 随机森林景观预测模型设计

在环境辅助决策链路中，系统根据既有材料的技术路线，引入 Python Flask 封装的随机森林预测服务，对云海、雾凇、冰挂等景观进行概率估计，并对日出日落、银河等场景提供联动分析。

从模型输入角度看，系统需要采集并组织气象与地理特征，包括湿度、露点差、云量、风速、降水、温度、海拔、小时天气趋势、日出日落时刻和光污染条件等信息。

这些特征会在后端被统一清洗与结构化，再作为特征向量发送到 Flask 服务端进行推理。

随机森林之所以适合本课题，是因为其能够处理多维度、非线性且存在噪声的环境特征，并对有限样本条件下的分类问题保持较好的鲁棒性。

在服务封装上，Flask 微服务负责模型加载、特征接收、概率计算和结果返回，Java 后端负责统一鉴权、路线上下文拼装、结果缓存和接口封装。

在结果输出上，预测服务不仅返回概率值，还应返回等级判断、最佳观赏时段、主要影响因素和风险提示，例如“云量偏高”“风速偏大”或“露点差不足”等内容。

同时，系统通过 `landscape_feature_snapshots` 和 `landscape_observation_labels` 两类数据表保存模型特征快照与人工观测标签，为后续模型训练、参数调优和效果验证提供数据基础。
