# 3 系统需求分析

## 3.1 可行性分析

从技术可行性来看，本课题采用的前后端分离架构与主流开发技术已经具备较高成熟度。

前端部分以 Vue、TypeScript 和 Vite 为基础，能够较好地支持组件化开发、页面路由管理、状态共享和复杂交互实现。

后端部分采用 Spring Boot 3、MyBatis-Plus、MySQL 和 Redis 等主流技术，能够满足用户管理、路线管理、评论互动、上传服务、后台治理和数据查询等常规业务需求。

同时，任务书和开题报告中提出的 DeepSeek、Function Calling、高德地图 API 以及 Python Flask 随机森林预测模型，也为系统向智能推荐和景观辅助决策方向扩展提供了明确技术路线。

从当前项目实际情况来看，仓库中已经形成前端、后台和后端服务的分工结构，并沉淀了页面组织、接口管理、上传能力、AI 会话链路、天气服务和景观预测等实现基础，这进一步说明本课题具备较强的工程落地条件。

从经济可行性来看，本系统依赖的大部分开发框架与基础软件均可基于开源生态完成构建。

前端框架、后端框架、数据库、中间件以及测试工具链都具有较低的使用门槛和较成熟的社区支持。

项目的主要成本集中在服务器资源、数据库部署、缓存服务、地图与天气接口调用以及大模型服务调用等方面。

通过本地数据库优先匹配、缓存复用和热点数据预计算等策略，可以减少对大模型与外部接口的高频调用，从而控制总体成本。

因此，从系统原型构建和毕业设计实现的角度出发，本课题具备较好的经济可行性。

从操作可行性来看，平台面向的用户群体主要是户外爱好者、徒步新手以及有路线分享需求的社区用户。

这类用户对路线、天气、装备和交流功能有较高需求，但未必具备专业 GIS 操作能力。

因此，系统需要通过地图可视化、卡片化路线展示、自然语言交互和结构清晰的个人中心来降低使用门槛。

当前论文主线与项目实现均围绕这一方向展开，说明系统在用户操作层面具有较好的可接受性和可实施性。

综合分析可知，本课题在技术、经济和操作三个层面均具备开展研究与实现的现实基础。

## 3.2 功能需求分析

### 3.2.1 用户模块需求

用户模块是整个平台运行的基础。

系统需要支持用户注册、登录、退出、身份识别和基本资料维护等常见功能，以满足平台个性化服务与社区互动的前提条件。

除基础账号能力外，平台还应支持用户画像信息维护，包括用户的户外经验、偏好的路线风格、轻装或重装倾向以及所在地区等内容。

这些信息既影响路线推荐的准确性，也决定了系统在智能交互过程中生成建议的贴合程度。

在当前项目内容中，个人中心、账号设置和徒步画像相关页面已经形成较清晰的组织结构，这为本模块的论文写作提供了较好的实现支撑。

### 3.2.2 路线与天气模块需求

路线模块承担着平台最核心的业务职责。

系统需要支持路线的列表查询、条件筛选、详情查看、路线发布、路线编辑、收藏、点赞和图文展示等功能。

在路线详情中，除名称、难度、距离、时长、海拔和标签等基础信息外，还应重点展示路线轨迹、位置环境、天气情况和景观提示等与户外决策密切相关的内容。

天气模块需要与路线模块形成联动。

当用户查看某条路线时，系统应能够提供目标区域的天气信息、未来时段预报、日出日落和月相等辅助信息。

如果系统进一步结合景观预测能力，对云海、日出、雾凇等户外关注度较高的场景进行概率提示，将更有利于提升平台的决策支持价值。

### 3.2.3 智能交互模块需求

智能交互模块是本课题的重要差异化功能。

系统需要支持用户以自然语言表达路线需求，并由 AI 能力对这些需求进行理解和结构化分析。

例如，用户可以直接输入“推荐一条适合周末新手徒步、风景好、最好能看日出的路线”，系统需要识别其中的地点约束、时长约束、难度偏好、负重偏好和景观偏好，再进一步联动路线检索与推荐能力生成结果。

在交互体验上，平台还应支持对话上下文管理、推荐结果插入、追问与二次筛选等行为，从而形成比普通搜索框更自然、更灵活的使用方式。

结合当前项目情况，聊天页面、会话列表、消息流、流式返回和路线卡片嵌入等工程组织已经具备补充到论文实现章节中的条件。

### 3.2.4 社区互动模块需求

社区互动模块用于增强平台的内容沉淀能力和用户粘性。

系统应支持用户发布图文路线体验、发表评论、进行点赞和收藏，并在社区页面中浏览其他用户分享的内容。

通过这一模块，平台可以逐步积累路线经验、景观照片、出行建议和评价反馈，形成具有户外特色的用户共建生态。

社区互动不仅能提升用户活跃度，也能为后续的智能推荐提供更丰富的数据基础。

当前项目中已经存在社区页面、评论接口和相关交互状态管理内容，这些都可以作为论文中本模块实现分析的补充依据。

### 3.2.5 系统管理员模块需求

在现有论文主线中，系统管理员模块承担平台内容治理与后台维护职责，因此本模块必须继续保留。

随着路线发布、评论互动和图片上传等能力不断增强，平台需要具备对用户内容进行审核、管理和维护的后台能力。

管理员应能够查看用户信息、审核路线内容、处理评论异常、维护系统配置项，并在必要时对平台中的风险内容进行控制。

从当前仓库情况看，后端已经存在管理员控制器、管理员相关数据对象、路线审核与治理接口以及后台配置能力，这些内容可以作为对论文中管理员模块实现的补充说明。

## 3.3 数据流图分析

### 3.3.1 用户路线检索与浏览数据流

图位预留：图 3-1 用户路线检索与浏览数据流图。

在用户路线检索与浏览场景中，数据流的输入源主要包括用户在首页和搜索页输入的关键字、难度、距离、时长、轻重装偏好以及地理位置筛选条件。

处理节点首先是前端页面表单与状态管理模块，它负责收集查询条件并向后端发送路线检索请求。

后端控制器接收请求后，将参数交由路线服务层进行统一解析，再结合数据库中的路线主表、标签表、点赞收藏关系表和评论统计信息完成综合查询。

数据存储层主要依赖 MySQL 中的 `trails`、`trail_tags`、`tags`、`trail_likes`、`trail_favorites` 和 `reviews` 等表，并可辅以 Redis 中的热门路线缓存和筛选结果缓存提升响应效率。

结果输出阶段由后端将分页路线数据封装为统一视图对象返回前端，前端进一步将结果渲染为路线卡片、筛选列表和详情入口，用户点击某条路线后再进入详情页继续浏览天气、轨迹、景观预测与评论信息。

### 3.3.2 路线发布与轨迹上传数据流

图位预留：图 3-2 路线发布与轨迹上传数据流图。

在路线发布与轨迹上传场景中，输入源主要包括用户在发布页填写的路线标题、地点、难度、时长、路线说明、图片资源以及 GPX 或 KML 轨迹文件。

处理节点首先由前端发布页完成表单校验、图片上传和轨迹文件上传请求发起，随后后端上传服务为媒体资源生成上传凭证或分片上传状态，并在上传完成后回写媒体文件记录。

当用户正式提交路线时，后端路线服务会整合表单字段、上传后的图片资源、轨迹解析结果以及结构化地理信息，生成待审核的路线记录。

数据存储层涉及 `media_files`、`upload_checkpoints`、`trails`、`trail_images` 和 `trail_tracks` 等表，其中轨迹表保存标准化 GeoJSON、起终点坐标、边界框、距离和海拔等关键数据。

结果输出阶段由系统向前端返回“路线已提交审核”的反馈信息，同时在管理员侧形成待审核记录，供后台审核流程进一步处理。

### 3.3.3 社区评论与互动数据流

图位预留：图 3-3 社区评论与互动数据流图。

在社区评论与互动场景中，输入源包括用户在社区页或路线详情页提交的评论文本、评分、回复对象、评论配图，以及用户发起的点赞和收藏操作。

处理节点由前端互动组件收集输入后调用评论服务或路线交互服务，后端再根据用户身份、目标路线和评论层级关系完成权限校验、业务处理和状态更新。

数据存储层主要依赖 `reviews`、`review_images`、`trail_likes`、`trail_favorites` 和 `trails` 等表，其中评论治理状态、点赞计数与收藏计数会随着用户操作同步更新。

结果输出阶段表现为评论列表刷新、点赞收藏状态即时变化、评论数与互动计数同步展示，同时后台还能够基于这些数据进行评论隐藏、恢复和异常内容治理。

### 3.3.4 AI 对话推荐与结果返回数据流

图位预留：图 3-4 AI 对话推荐与结果返回数据流图。

在 AI 对话推荐场景中，输入源主要是用户在 AI 助手页输入的自然语言需求，以及当前用户画像、历史会话上下文和前端附带的城市或路线上下文信息。

处理节点首先由前端会话模块将消息发送到后端 AI 服务，后端完成会话创建或会话续接后，将用户输入写入 AI 消息表，并交由大模型语义理解模块解析地点、难度、时长、景观偏好和检索意图。

在此基础上，系统通过 Function Calling 机制或等价的结构化调用链，将非结构化输入转换为内部可执行的路线检索参数，再结合路线库、用户画像和热度评分完成候选路线筛选与排序。

数据存储层主要包括 `ai_conversations`、`ai_messages`、`user_hiking_profiles`、`trails` 以及与路线相关的标签、互动和环境数据表，同时还可以通过 Redis 暂存会话上下文和热门推荐结果。

结果输出阶段由后端通过 SSE 或 WebSocket 持续回传流式文本、路线卡片和追问建议，前端将这些内容嵌入消息流中展示，从而形成自然语言对话与真实路线对象联动的推荐体验。

## 3.4 非功能需求分析

除功能需求外，系统还必须满足一定的非功能需求，以保证平台在真实使用场景中的可用性与稳定性。

首先是性能需求。

平台需要在常见的路线检索、详情查看、评论加载和聊天交互场景下保持较快响应速度。

在路线列表、首页热点内容、后台统计和用户常用数据访问过程中，通过数据库索引优化与 Redis 缓存策略降低重复查询开销，是提高平台性能的重要手段。

其次是安全性需求。

系统涉及用户账号、个人资料、发布内容、会话记录和后台治理操作等敏感信息，因此需要通过登录认证、接口权限控制、参数校验、上传约束和异常处理等方式提升安全保障能力。

在设计上，引入 Spring Security 和 JWT 机制，能够为系统提供较为清晰的访问控制基础。

再次是可维护性需求。

毕业设计论文不仅强调功能完成，也强调工程组织和后续维护能力。

系统应尽量采用分层设计、模块化实现和统一的数据对象组织方式，以便后续增加新页面、新接口和新模型能力。

当前仓库采用 monorepo 方式组织前后端代码，并将控制器、服务层、数据对象和页面组件进行分目录管理，这对系统维护具有积极作用。

最后是可扩展性需求。

由于户外场景具有明显演进空间，后续系统还可能继续扩展图片识别定位、更多景观预测类型、后台数据看板和多源天气服务等能力。

因此，论文中的系统设计必须保留足够扩展空间，使平台能够在既有基础上逐步演化。
