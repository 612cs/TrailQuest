## 英文摘要

### 可直接替换版

ABSTRACT

In recent years, hiking, mountaineering, and camping have become common outdoor activities for many users. Before travelling, users often need to check route distance, elevation gain, difficulty, weather, track data, and comments from other users. However, this information is usually scattered across map tools, social platforms, and travel notes. This makes route selection time-consuming and increases the uncertainty of outdoor planning.

This thesis designs and implements an intelligent outdoor route recommendation platform based on Vue. The system uses a front-end and back-end separated architecture. The front end is developed with Vue, TypeScript, Tailwind CSS, and DaisyUI, while the back end uses Spring Boot, MyBatis-Plus, MySQL, Redis, and Spring Security to provide API services, data management, and permission control. In the recommendation module, the DeepSeek large language model and Function Calling are used to parse natural language input into structured conditions such as location, difficulty, duration, and equipment preference. The back end then retrieves candidate routes, ranks them, and returns route cards through a WebSocket message channel. For environmental assistance, the system combines Amap API, weather data, and a random forest model to predict the probabilities of sea of clouds, rime, icicles, and sunrise or sunset scenery.

The test results show that the platform can support user authentication, route search, route publishing, community interaction, AI recommendation, scenery prediction, and administrative review. Under the test conditions of this graduation project, the system can provide users with more concentrated route and environmental information before outdoor travel.

KEYWORDS: Outdoor Route Recommendation; Vue 3; Spring Boot 3; DeepSeek Large Language Model;

## 1 绪论

### 替换稿

#### 1.1 研究背景

徒步、登山、露营和越野跑等户外活动近几年持续升温。对普通用户来说，选择一条合适路线并不只是查找目的地，还要了解路线长度、累计爬升、路面情况、天气变化、装备要求和实际体验。特别是在山地、峡谷、森林步道等场景中，路线信息往往比城市道路导航更复杂，单一地图工具难以覆盖全部判断因素。

当前用户获取户外路线信息的渠道较分散。一部分信息来自地图应用，一部分来自社交平台、游记、轨迹文件和评论区。地图工具更擅长道路导航和位置检索，但对非铺装路线、高程变化、局地天气和景观窗口的表达有限。社交平台中的内容更接近真实体验，却常常缺少统一结构，用户需要花较多时间筛选和对比。由此带来的问题是，路线选择依赖个人经验，风险判断也容易受到信息不完整的影响。

基于上述情况，本文将研究对象限定为中文户外路线推荐平台。系统尝试把路线数据、地图轨迹、天气信息、社区评价、AI 对话和景观预测放在同一平台中处理，使用户在出发前能够更集中地完成路线查询、比较和判断。

#### 1.2 研究意义

本课题的意义主要体现在系统实现和应用服务两个方面。

在系统实现方面，本文尝试将 AI 语义理解与 GIS 路线数据结合起来。大语言模型适合处理用户自然语言中的地点、时间、难度、景观偏好和装备要求，但路线筛选、权限校验、距离统计和结果排序仍需要后端服务完成。基于这一特点，系统采用“本地候选路线检索 + 大模型按需解析”的方式。用户输入先由 DeepSeek 进行语义解析，解析结果再交给后端查询路线库并生成推荐卡片。这样既保留了自然语言交互的便利性，也避免把所有业务判断都交给大模型。

在应用服务方面，平台将路线画像、用户画像和天气景观信息共同用于推荐。用户可以通过搜索页进行条件筛选，也可以在 AI 助手中用接近日常表达的方式提出需求。系统根据难度、时长、距离、负重偏好、天气情况和景观概率组织推荐结果，并给出简要理由。对户外出行而言，这类说明比单纯的热度排序更有参考价值。

此外，天气与景观预测模块为路线详情和 AI 推荐补充了环境信息。系统基于天气数据和随机森林模型预测云海、雾凇、冰挂、日出日落等景观概率，并提示降雨、低能见度等风险。该设计能够帮助用户在路线选择之外进一步考虑出行时间和安全因素。

#### 1.3 国内外研究现状

#### 1.3.1 国外研究现状

国外关于路线规划、空间信息服务和个性化推荐的研究起步较早。早期研究多关注最短路径和通行效率，近年的研究则开始考虑安全性、舒适性、景观性和用户偏好。Siriaraya 等对质量感知步行导航进行了综述，指出路线推荐不能只依赖距离，还应考虑环境体验、安全性和可达性[1]。在旅游推荐领域，Zhang 从大数据角度讨论智能旅游路线推荐，Liu 和 Wang 分别将知识图谱、知识映射等方法引入旅游路线推荐研究[2-4]。Su 等和 Zheng 等的研究进一步说明，知识图谱、多目标优化等方法可用于处理旅游场景下的多约束排序问题[19-20]。

大语言模型出现后，推荐系统开始引入自然语言交互和工具调用能力。Wang 等提出 ToolFlow，用合成工具调用对话提升模型在多步骤任务中的执行能力[8]。Qin 等研究开放域 Function Calling 能力，为大模型连接外部工具和业务系统提供了参考[9]。这些研究与本文的 AI 推荐模块关系较密切：用户并不总是用标准关键词表达需求，系统需要先理解自然语言，再把需求转换为后端可以执行的查询条件。

推荐结果能否被用户接受，也与解释方式有关。Li 等从 Quantified Self 视角研究用户对 AI 推荐的接受机制，指出用户画像、可量化信息和推荐解释会影响用户信任[17]。户外路线推荐也存在类似问题。用户通常要在体力、时间、天气、景观和安全之间取舍，因此系统需要给出路线卡片、推荐理由和追问建议，而不是只返回一个排序列表。

在环境预测方面，Kim 等使用随机森林方法开展区域能见度预测，说明机器学习模型可以利用气象要素判断低能见度场景[10]。Lakra 和 Avishek 对雾的形成、分类和预测进行了综述，指出湿度、温度、风速和气溶胶等因素都会影响雾和低能见度天气[11]。Parde 等提出雾概率预报决策支持系统，为户外出行风险提示提供了方法参考[21]。这些研究为本文采用天气数据、地理环境和随机森林模型进行景观概率预测提供了依据。

从国外研究可以看出，路线推荐、AI 工具调用和气象预测已经分别形成了较多成果。但这些研究通常分布在导航、旅游推荐、语言模型和气象预测等不同方向，直接面向中文户外徒步场景，并同时接入 GIS 地图、用户画像、AI 对话和景观预测的系统研究仍不多。

#### 1.3.2 国内研究现状

国内智慧旅游和地图服务发展较快，相关研究已经从景点信息展示逐步扩展到平台服务、用户参与和智能推荐。符琳蓉、汪明峰从服务生态系统角度讨论智慧旅游价值共创，认为智慧旅游平台需要处理多主体协作与数据连接问题[5]。周景兰从网络平台服务模式角度分析智慧旅游建设，强调资源整合、在线服务和用户体验的持续优化[14]。这些研究说明，国内智慧文旅建设已经开始关注平台化运营和用户服务链路。

在工程实现方面，Spring Boot 和 Vue 是国内旅游服务系统中较常见的组合。肖程鸣、曾志颖实现了基于 Spring Boot 和 Vue 的红色智慧旅游平台[6]，黄镓升、邓舒婷完成了基于 Spring Boot 的南宁旅游 APP 设计[13]，曹浩等设计了基于 SpringBoot+Vue 的桂林龙胜各族自治县旅游信息系统[22]。这些系统在旅游资源展示、用户管理、路线信息维护和后台管理方面与本文系统具有相似之处，也为本文的技术选型提供了工程参考。

在乡村文旅和路线推荐方向，李晟曈等实现了基于 Vue 和 SpringBoot 的乡村文旅平台，系统通过二维码标识、景点详情、精确搜索和信息推荐整合乡村文旅资源[23]。戴亚哲等在 SpringBoot+Vue 文旅平台中引入粒子群优化算法进行旅游景点路线推荐[24]。这些研究表明，国内文旅系统已经开始将推荐算法与平台功能结合起来，但面向户外徒步路线的轨迹解析、环境风险和景观窗口支持仍相对有限。

地图服务和智能推荐也是本文需要借鉴的方向。陈慕花总结了计算机技术在智慧旅游数据处理、可视化和智能服务中的作用[7]。乔振华、王珺祺基于百度地图完成吉林省智慧旅游地图设计，说明地图 API 与旅游资源数据结合可以提升路线展示和位置引导能力[16]。王娜、何文静、孙倬研究大学生面对智能推荐算法时的应对行为，提示推荐系统应关注用户对结果的理解与信任[12]。马玉彬等将贝叶斯优化随机森林用于储层预测，体现了随机森林在复杂特征组合和非线性关系拟合中的应用价值[15]。

国内已有研究为本文提供了前后端分离开发、地图接入、路线推荐和平台管理方面的参考。不过，现有系统多面向城市旅游、景区服务或通用文旅平台，对户外徒步场景中的轨迹文件、爬升、动态天气、景观概率和社区治理关注不足。本文的工作重点正是把这些分散功能放入一个可运行平台中进行整合。

#### 1.3.3 现状综述

国内外研究已经分别在路线推荐、智慧旅游平台、地图服务、AI 工具调用和环境预测等方向形成基础。国外研究更关注质量感知路线导航、大语言模型工具调用和气象预测模型[1-4,8-11,17,19-21]；国内研究则更多围绕智慧文旅平台建设、前后端分离系统实现、地图服务接入和旅游推荐功能展开[5-7,12-18,22-24]。

这些成果对本文有直接参考价值，但应用到户外徒步场景时仍存在不足。很多系统以景区游览或城市道路为主要对象，对高山、峡谷、森林等自然环境中的轨迹解析、天气变化和景观窗口支持不够。部分平台虽然提供推荐功能，但主要依赖标签匹配、热度排序或静态筛选，对用户自然语言需求和体力偏好的理解较弱。AI 对话、GIS 地图、天气服务、景观预测和社区治理也常被分开研究，缺少面向完整出行决策过程的系统实现。

因此，本文把研究范围收束到户外路线推荐平台。系统不是单独验证某一种算法，而是在一个前后端可运行的应用中，把路线数据组织、地图轨迹展示、自然语言推荐、天气与景观辅助判断、评论互动和后台审核串联起来，观察这些能力如何共同服务于用户的路线选择。

#### 1.4 本文的主要工作与论文结构

本文围绕户外路线智能推荐平台的设计与实现展开，主要完成以下工作。

第一，整理户外路线平台的业务需求，划分普通用户、管理员和智能服务三类功能。普通用户侧包括路线浏览、路线发布、评论互动、个人中心和 AI 推荐；管理员侧包括路线审核、内容治理、用户管理和操作日志；智能服务侧包括自然语言推荐、天气查询和景观概率预测。

第二，完成系统总体设计。系统采用前后端分离架构，前端基于 Vue 3 构建用户端与管理端页面，后端基于 Spring Boot 提供接口服务和权限控制，数据库围绕用户、路线、评论、媒体、AI 会话和景观预测等实体组织数据。

第三，实现路线推荐与景观预测相关功能。AI 推荐模块使用 DeepSeek 和 Function Calling 解析用户需求，并通过 WebSocket 返回消息流和路线卡片；景观预测模块结合天气数据和随机森林模型，输出云海、雾凇、冰挂、日出日落等景观概率。

第四，开展系统测试。测试内容覆盖认证、路线检索、路线发布、社区互动、AI 推荐、后台审核、媒体上传和性能表现，验证主要业务链路是否能够完成。

## 4 系统总体设计

### 替换稿

#### 4.1 系统架构设计

本系统按照前端展示、后端服务和 AI/算法服务三个层次组织。前端负责页面交互和数据展示，后端负责接口鉴权、业务处理和数据持久化，AI/算法服务负责自然语言解析、路线推荐和景观概率预测。这样的划分能够把页面逻辑、业务规则和智能计算分开，便于后续定位问题和维护模块。

图4.1 系统架构图

#### 4.1.1 前端展示层

前端展示层由 Vue 3、Pinia 和浏览器运行环境组成。用户看到的路线列表、地图轨迹、评论列表、AI 对话结果和后台页面，都在这一层完成渲染。前端不直接处理复杂业务规则，而是把用户操作转换成接口请求，再把后端返回的数据组织成页面状态。

在组件组织上，系统使用 Vue 3 的 Composition API 拆分页面逻辑。路线卡片、筛选面板、评论列表、上传表单和地图容器等功能被拆成相对独立的组件，便于在用户端和管理端复用。状态管理由 Pinia 承担，主要保存登录令牌、用户信息、路线互动状态和 AI 会话上下文。

地图展示使用高德地图 API。后端返回的 GeoJSON 轨迹数据会在前端渲染为路线折线，并配合起终点标记、路线距离、爬升和天气信息展示。对于照片墙和评论列表等数据量较大的页面，前端采用列表按需渲染，减少一次性创建过多 DOM 节点带来的页面压力。

AI 推荐页面通过 WebSocket 接收后端推送的消息流。模型回复和路线卡片不会等所有内容生成完成后一次性展示，而是在对话过程中逐步挂载到页面中，使用户能够更早看到推荐结果和追问提示。

图4.2 前端展示层架构图

#### 4.1.2 后端服务层

后端服务层以 Spring Boot 为核心，连接前端请求和数据库资源。它主要处理用户认证、路线查询、路线发布、评论互动、文件上传、后台审核和操作日志等业务。前端提交的请求会先经过认证和参数校验，再进入对应控制器和服务层。

权限控制采用 JWT 和 Spring Security 实现。普通用户登录后可以访问路线浏览、评论、收藏、路线发布和个人中心等接口；管理员登录后可以进入后台，对路线、用户和评论进行审核或治理。系统通过角色判断限制后台接口，避免普通用户访问管理功能。

数据持久化使用 MySQL。用户、路线、评论、媒体文件、AI 会话、AI 消息、审核日志和景观预测相关数据都以结构化表的形式保存。Redis 只承担短期状态校验作用，主要用于 OSS 上传凭证和对象键的状态检查。点赞、收藏、审核等会改变业务状态的操作，仍然以数据库事务和服务层校验保证一致性。

图4.3 后端服务层架构图

#### 4.1.3 AI 服务层

AI 服务层处理普通业务接口之外的智能计算任务，主要包括自然语言推荐和景观预测。它不直接替代后端业务逻辑，而是为后端提供结构化参数、推荐解释和环境辅助信息。

在 AI 推荐中，DeepSeek 大语言模型用于理解用户输入。用户可能输入“周末想找一条新手轻装路线”这类自然语言，系统需要从中提取地点、时间、难度、行程天数、装备偏好和景观偏好等条件。Function Calling 将这些条件约束为后端能够识别的参数，后端再查询路线库、排序候选路线，并生成路线卡片。

景观预测由 Spring Boot 后端中的预测服务完成。系统根据路线坐标和轨迹信息获取天气、天文、光污染和海拔等上下文数据，构造温湿度、风速、云量、降水、露点差等特征，再输入随机森林预测器和规则修正逻辑，输出云海、雾凇、冰挂、日出日落等景观的概率、等级和风险提示。

#### 4.2 系统功能设计

平台功能分为前台用户、后台管理员和智能服务三类。

前台用户侧主要面向户外爱好者。用户可以浏览路线、搜索路线、查看路线详情、上传轨迹文件、发布路线、参与评论互动，也可以在个人中心维护基本资料和徒步画像。路线详情页集中展示封面、相册、轨迹、天气、景观预测和评论，帮助用户完成出行前的信息判断。

后台管理员侧主要用于平台内容治理。管理员可以查看待审核路线，结合文本、图片和轨迹信息决定通过或驳回；也可以对用户、评论和已发布路线进行治理，并查看后台操作日志。该部分功能保证用户生成内容进入平台前经过基本审核。

智能服务侧包括 AI 推荐和环境辅助判断。AI 推荐根据用户自然语言需求生成结构化查询条件，并结合用户画像和路线数据返回推荐卡片。天气与景观预测模块基于路线位置和外部天气数据，给出未来天气、景观概率和风险提示。三类功能共同支撑平台从“路线展示”扩展到“路线选择辅助”。

#### 4.4 AI 大模型与算法集成设计

本系统的智能服务由两条链路组成：一条用于自然语言路线推荐，另一条用于天气与景观预测。

在路线推荐链路中，用户输入先进入 AI 对话服务。大模型根据提示词和 Function Calling 规则抽取地点、时间、难度、装备偏好、景观偏好等参数。后端收到结构化参数后，不直接采用模型生成的路线结论，而是从本地路线库中检索候选路线，并结合路线属性、用户画像和环境信息进行评分。最终结果以文本说明和路线卡片的形式返回前端。

在景观预测链路中，系统先根据路线坐标获取天气和环境上下文，再构造特征输入随机森林预测器。预测结果包括云海、雾凇、冰挂、日出日落等景观的概率、等级和风险说明。该结果既用于路线详情页展示，也可作为 AI 推荐中的辅助特征。当用户希望寻找“适合看云海”或“日出概率较高”的路线时，系统能够把景观预测结果加入推荐理由。

这两条链路共同完成自然语言理解、路线检索、环境判断和结果解释。大模型主要承担意图解析和表达生成，路线筛选、评分、权限校验和数据落库仍由后端服务控制。

## 7 结论

### 可直接替换版

#### 7.1 全文总结

本文围绕户外路线发现、路线发布、社区互动、后台审核和智能推荐等需求，完成了一个基于 Vue 3 与 Spring Boot 的户外路线智能推荐平台。系统采用前后端分离架构，前端实现路线浏览、地图展示、轨迹上传和 AI 对话交互，后端实现用户认证、路线管理、评论互动、媒体存储和审核日志等业务。

在智能服务方面，系统将 DeepSeek 大语言模型用于自然语言意图解析和推荐说明生成，将 Function Calling 用于约束结构化参数输出，并通过 WebSocket 向前端返回消息流和路线卡片。景观预测模块基于天气数据和随机森林模型，对云海、雾凇、冰挂和日出日落等景观概率进行预测，并为路线详情和 AI 推荐提供环境辅助信息。

测试结果表明，系统能够完成认证、路线查询、路线发布、评论互动、AI 推荐、景观预测和后台审核等主要流程。整体实现满足毕业设计阶段的验证要求，也为后续扩展真实路线数据和用户反馈打下了基础。

#### 7.2 存在的问题与未来展望

当前系统仍存在一些不足。首先，路线数据和真实用户反馈数量有限，推荐结果更多依赖已有路线字段、用户画像和规则评分。后续可以在获得用户授权的前提下，继续积累路线浏览、收藏、评价和出行反馈数据，用于改进推荐排序。

其次，景观预测模块已经实现云海、雾凇、冰挂和日出日落概率输出，但历史样本规模仍不足。后续可以接入更长时间跨度的天气、地形和实地观测样本，并使用准确率、召回率、AUC 等指标对模型效果进行更系统的评估。

最后，平台目前主要以文字、图片、轨迹和天气数据为基础。随着用户上传内容增加，后续可以尝试引入图像识别或多模态分析方法，对路线图片中的地貌、路况和风险特征进行识别，并把这些信息补充到路线画像中。相关扩展需要建立在数据质量和权限合规的基础上逐步推进。
