# 5 系统详细设计与实现

## 5.1 开发环境搭建与项目初始化

系统开发环境围绕前后端分离架构展开组织。

在工程实现上，项目采用 monorepo 方式管理多个应用模块，将用户前端页面系统放在 `apps/web` 中，将后台管理端放在 `apps/admin` 中，将 Spring Boot 后端服务放在 `apps/api` 中。

这种组织方式便于统一管理依赖、脚本命令、接口联调和后续测试流程，也有利于在论文中明确区分前台用户界面、后台管理界面和服务端能力之间的边界。

前端部分通过 Vite 提供开发服务器与构建能力，后端部分通过 Maven 管理依赖并运行 Spring Boot 服务。

从命令组织来看，项目已经提供前端启动、后台启动、后端启动、前端构建、后端构建、前端单元测试、前端端到端测试和后端测试等脚本入口。

这说明系统已经具备较为完整的基础开发环境与工程化运行条件。

## 5.2 前端核心模块实现

### 5.2.1 首页实现

首页是平台对外展示的第一入口，主要负责整合热门路线、主题活动入口和首页运营内容。

在当前项目中，首页已经围绕大图引导区、活动入口、热门路线卡片和快捷导航等内容展开组织。

用户进入系统后，可以在首页快速完成路线发现、活动浏览和功能跳转，这种设计能够降低初次使用门槛，也符合户外平台“先看内容、再决定是否深入筛选”的使用习惯。

结合后台设置与配置中心能力，首页中的部分展示内容还支持动态维护，使运营信息不必完全依赖硬编码方式更新。

### 5.2.2 搜索页实现

搜索页是平台承接路线检索需求的核心页面。

该页面通过关键字输入与多条件筛选相结合的方式，支持用户按照难度、距离、时长、轻重装偏好和地理范围等维度缩小结果范围。

在交互实现上，搜索页需要在筛选条件改变后及时刷新列表，并将当前条件状态保持在页面层和路由层之间，以便用户在来回切换页面时仍能保留检索上下文。

搜索结果页中的路线卡片直接承接到路线详情页，从而形成“搜索筛选 - 结果浏览 - 详情查看”的典型前端闭环。

### 5.2.3 社区页实现

社区页主要用于展示用户发布的路线体验、图文内容和互动反馈，是平台沉淀 UGC 内容的重要页面。

当前项目中，社区页已经形成内容卡片、互动状态展示和基础流式浏览结构。

用户可以在该页面浏览其他用户的徒步记录，并通过点赞、收藏、评论等方式参与互动。

社区页的价值不仅在于增强平台活跃度，也在于为路线详情页之外提供另一种“从真实体验反推路线价值”的浏览入口。

### 5.2.4 路线详情页与地图展示实现

路线详情页是本系统信息最集中的页面之一。

页面需要同时展示路线名称、地点、难度、距离、海拔、时长、封面图、作者信息、轨迹数据、天气信息、景观提示和评论内容。

在技术路线中，高德地图 API 是路线空间可视化的重要基础。

通过地图服务能力，系统能够将路线轨迹、起终点和位置提示以更直观的方式呈现给用户。

从当前项目来看，路线详情页已经具备路线轨迹展示、天气加载、景观预测卡片展示和评论模块组织等实现基础。

用户在详情页内不仅可以完成路线信息阅读，还可以执行点赞、收藏、评论、跳转图片展示页等操作，使详情页真正成为路线消费与互动的中心节点。

### 5.2.5 个人中心页实现

个人中心页主要用于展示用户资料、发布内容、收藏内容和个人配置项。

在当前项目工程中，个人中心页面已经围绕标签页切换、资料编辑、徒步画像配置和内容管理展开实现。

用户可以在此查看自己发布过的路线、收藏过的路线，并完成昵称、头像、简介、所在地和徒步画像等信息维护。

这一页面使系统在个性化管理层面形成了较为稳定的前端结构，也为 AI 推荐服务获取用户偏好提供了来源。

### 5.2.6 发布页实现

发布页对应平台中的路线生产能力，是路线从“个人经验”向“结构化平台内容”转化的关键页面。

用户可在该页面填写路线名称、地点、难度、时长、距离、路线描述等基础信息，并上传封面图、相册和轨迹文件。

从仓库实现情况看，发布页已经形成图像预览、地图容器、轨迹解析、位置提示和天气辅助解析等能力。

在交互逻辑上，发布页会先完成本地表单校验，再协同上传服务完成媒体资源处理，最后将结构化表单数据与轨迹解析结果一并提交给后端路线服务。

### 5.2.7 AI 助手页实现

AI 助手页是系统智能交互能力的前端承载界面。

根据现有论文和开题报告的主线，该页面需要支持用户通过自然语言查询路线，并由系统给出个性化推荐结果。

从当前项目实现来看，聊天页面已经形成会话列表、消息气泡、推荐问题、追问流程、消息流底部滚动和错误提示等交互结构。

同时，系统已经具备将推荐路线以卡片形式嵌入对话流中的前端能力，这使 AI 推荐结果不再只是纯文本，而能够与真实路线对象形成联动展示。

这种页面组织方式显著提升了推荐结果的可读性，也增强了用户对推荐内容的点击与转化效率。

### 5.2.8 路线图片展示页实现

路线图片展示页用于承接路线详情页中的图片浏览需求，使路线相册具备更强的沉浸式阅读效果。

当用户在详情页中点击相册内容后，可以进入单独的图片展示页面，以更清晰的方式查看路线沿途景观、地形特征和用户拍摄内容。

这一页面虽然不改变平台主业务逻辑，但在视觉层面强化了路线内容表达，也使社区分享和路线详情展示更加完整。

### 5.2.9 Admin 系统页面实现

后台管理端作为独立应用存在，采用单独路由与壳层布局组织页面。

前端通过管理员登录、权限校验和页面路由守卫，确保后台功能只在管理员身份下可见。

#### 5.2.9.1 后台登录页

后台登录页用于管理员身份验证。

系统在登录成功后写入后台登录状态，并在后续页面访问中通过路由守卫校验管理员权限，从而阻断普通用户进入后台。

#### 5.2.9.2 后台首页

后台首页用于展示平台总体运行情况，包括路线审核统计、治理提醒、待处理事项和快速入口等内容。

该页面帮助管理员在进入后台后快速掌握平台状态，也为高频治理动作提供统一入口。

#### 5.2.9.3 操作日志页

操作日志页用于按时间和对象维度展示后台治理记录。

管理员可以查看操作模块、动作类型、目标对象、处理结果和操作时间，从而增强后台治理的可审计性。

#### 5.2.9.4 路线审核列表与详情页

路线审核列表页主要展示待审核路线，支持分页浏览和状态筛选。

进入审核详情页后，管理员可以查看路线基础信息、图文内容、轨迹情况和提交信息，并执行通过或驳回操作。

#### 5.2.9.5 路线管理列表与详情页

路线管理页面向已进入平台的路线内容，支持下架、恢复和详情复核等动作。

与审核页相比，路线管理更强调上架后治理与运营管理，是后台“内容存量治理”的主要入口。

#### 5.2.9.6 用户管理页

用户管理页用于展示用户列表、角色、账号状态和必要的治理入口。

管理员可以在此执行封禁与解封操作，并结合原因说明对异常用户进行控制。

#### 5.2.9.7 评论管理页

评论管理页用于查看评论内容、评论状态和对应路线信息。

管理员可以在该页面执行评论隐藏、恢复、删除和批量治理等动作，以维持社区内容质量。

#### 5.2.9.8 举报处理页

举报处理页用于集中展示需要人工核查的异常内容或举报事项。

管理员在核验信息后，可以执行处理完成动作，使平台形成从发现问题到治理落地的闭环。

#### 5.2.9.9 设置中心页

设置中心页主要承接首页大图等站点级运营配置维护。

通过该页面，管理员可以调整首页视觉展示内容，而无需改动前端代码。

#### 5.2.9.10 配置中心页

配置中心页用于维护系统选项组和选项项，包括徒步经验、路线风格、负重偏好、首页活动入口和搜索筛选项等。

该页面使前台页面中的部分枚举项具备后台可维护能力，增强了系统后续扩展性。

## 5.3 后端核心服务实现

### 5.3.1 用户认证与权限控制

后端通过 Spring Boot 组织控制器、服务层与数据访问层，完成用户、路线、评论、上传、配置和 AI 等主要业务模块的实现。

在认证机制方面，系统引入 Spring Security 与 JWT 方式完成登录态识别和接口访问控制。

用户通过 `POST /api/auth/login` 或 `POST /api/auth/register` 完成登录与注册，登录成功后获取访问令牌，前端在后续请求中携带该令牌访问受保护资源。

系统还提供 `GET /api/auth/me` 和 `GET /api/admin/me` 等接口，用于在前台和后台分别获取当前用户信息与权限状态。

这一机制为用户资料维护、路线发布、评论发表、AI 会话调用和后台治理等操作提供了安全保障。

### 5.3.2 路线发布服务

路线发布服务是平台内容生产链路的核心。

前端发布页在表单校验完成后，会调用 `POST /api/trails` 提交路线创建请求，后端接收后将基础信息、上传后的图片资源、轨迹定位信息和结构化地理字段统一整合为路线实体。

在路线修改场景下，系统通过 `PUT /api/trails/{id}` 完成重新提交流程，并保持“提交后进入审核”这一治理逻辑不变。

路线服务在落库时会同时维护 `trails`、`trail_images`、`trail_tags` 和 `trail_tracks` 等相关表，并对审核状态、创建时间、发布者信息和交互统计字段进行初始化或更新。

在与前端联动方面，服务端会返回完整的路线详情对象，供前端在提交成功后展示结果或跳转详情页使用。

### 5.3.3 轨迹上传与解析服务

轨迹上传与解析服务用于承接 GPX 或 KML 文件的上传和结构化处理。

该链路首先依赖上传服务完成轨迹原始文件保存，随后由轨迹解析服务对文件内容进行标准化处理，提取 GeoJSON、轨迹点数量、起终点坐标、边界框、距离、海拔和时长等信息。

解析结果会落入 `trail_tracks` 表，并与对应路线 ID 建立唯一关联。

这些数据不仅用于路线详情页地图展示，也为天气服务的精确定位、景观预测的特征构造和 AI 推荐中的地理解释提供关键支撑。

当前项目中，路线创建与更新接口已经支持绑定轨迹与结构化地理字段，这说明该服务链路已经具备较好的后端实现基础。

### 5.3.4 评论服务

评论服务负责路线详情页和社区互动链路中的评论数据处理。

系统通过 `GET /api/trails/{trailId}/reviews` 加载评论列表，通过 `POST /api/reviews` 发布评论，通过 `DELETE /api/reviews/{id}` 删除评论。

在业务处理过程中，系统会根据评论是否为顶级评论或回复，分别维护 `trail_id`、`parent_id`、`reply_to` 和评分等字段，并在必要时写入评论图片数据。

评论相关数据主要落入 `reviews` 和 `review_images` 表，同时也会反馈影响路线的评论数和详情页展示状态。

在与前端联动方面，评论服务需要支持评论即时刷新、回复关系展示和删除后状态同步，以保证社区互动体验的连续性。

### 5.3.5 上传服务

上传服务主要围绕头像、首页大图、路线封面、路线相册、轨迹文件和评论图片等媒体资源展开。

后端通过 `POST /api/uploads/sts` 为前端生成上传凭证，通过 `POST /api/uploads/complete` 在上传完成后写入媒体文件记录。

对于较大的媒体资源和轨迹文件，系统还通过上传断点机制维护分片上传过程，并将断点信息记录在 `upload_checkpoints` 表中。

所有已完成的媒体资源最终统一保存在 `media_files` 表中，并通过业务类型字段区分不同使用场景。

该服务与前端发布页、头像修改、评论配图和后台首页大图设置等页面均有直接联动，是平台实现完整多媒体能力的重要基础。

### 5.3.6 后台管理与配置服务

后台管理与配置服务负责承接管理员治理与运营维护需求。

在治理侧，系统通过 `GET /api/admin/dashboard/summary`、`GET /api/admin/operation-logs`、`GET /api/admin/trails`、`GET /api/admin/users`、`GET /api/admin/reviews` 和 `GET /api/admin/reports` 等接口向后台前端提供列表和统计数据。

在动作侧，系统通过路线审核通过与驳回、路线下架与恢复、用户封禁与解封、评论隐藏与恢复、评论删除和举报处理等接口完成治理动作执行，并同步记录后台操作日志。

在配置侧，系统通过 `GET/POST /api/admin/settings/home-hero` 维护首页大图，通过 `GET /api/admin/config/groups`、`GET /api/admin/config/groups/{groupCode}/items`、`POST /api/admin/config/groups/{groupCode}/items` 和 `PUT /api/admin/config/groups/{groupCode}/items/{itemId}` 维护系统选项组与选项项。

该服务在数据落库上主要依赖 `app_settings`、`system_option_groups`、`system_option_items` 和 `admin_operation_logs` 等表，并通过管理员权限控制和日志审计保障治理过程可控、可追踪。

## 5.4 智能推荐与预测服务实现

### 5.4.1 基于大模型的路线推荐实现

#### 5.4.1.1 自然语言输入与用户画像融合

在技术主线上，系统通过 DeepSeek 大模型和 Function Calling 机制完成自然语言理解与业务能力调用之间的衔接。

用户进入 AI 助手页后，可以直接输入诸如“帮我找一条周末单日、新手友好、风景好、最好能看日出的路线”这样的自然语言需求。

系统不会把这类输入当作普通搜索框关键字直接处理，而是先与用户画像、历史会话、当前位置和当前页面上下文一起组织成完整输入上下文。

用户画像中保存的经验等级、常走路线类型和负重偏好，会在推荐链路中参与意图解释和结果排序，使推荐结果兼顾“用户说了什么”和“用户一贯适合什么”两个维度。

#### 5.4.1.2 意图识别与参数结构化

在语义分析阶段，DeepSeek 大模型负责识别用户消息中的推荐意图、地点范围、难度要求、路线时长、距离范围、装备偏好和景观关键词。

经过识别后，系统将非结构化消息转换为结构化参数对象，包括地点文本、结构化省市区信息、难度标签、单日或多日类型、距离区间和场景标签等内容。

这种做法的优势在于，系统可以把模糊表达转化为后端可计算的检索条件，避免 AI 推荐完全依赖自由文本生成。

当大模型结构化结果不足时，系统还会通过规则解析和关键词兜底策略继续完成参数提取，从而保证推荐流程具有较好稳定性。

#### 5.4.1.3 Function Calling 调用策略

在完成参数结构化之后，系统通过 Function Calling 机制将模型理解结果映射为内部业务能力调用。

从论文实现角度看，这一阶段的核心工作是让模型知道“应该调用什么能力”，而不是让模型直接编造推荐结果。

系统可将内部业务能力抽象为路线检索、用户偏好获取、天气查询和景观预测查询等函数集合，并由后端统一处理鉴权、参数清洗和结果封装。

在当前工程中，这一链路已经能够完成会话创建、用户消息写入、推荐参数解析、路线库检索和候选卡片生成等关键步骤，说明 Function Calling 的业务协同思想已经得到落地支撑。

#### 5.4.1.4 路线候选筛选与排序

在候选路线生成阶段，系统首先根据地点、难度、时长、距离和装备条件查询路线库。

如果精确地理匹配结果不足，系统会自动执行位置文本回退检索，以提高召回率。

在排序阶段，系统综合考虑地理位置匹配程度、难度匹配程度、装备偏好匹配程度、用户画像适配度、路线评分、点赞数、收藏数和标签命中情况，对候选路线进行打分排序。

最终系统只保留最符合当前需求的若干条路线，并为每条路线生成推荐理由，例如“位置接近目标区域”“强度适合新手”“具备日出或云海相关景观条件”等。

这种排序方式使推荐结果既能体现大模型对语义的理解，又能落回真实可解释的业务数据之上。

#### 5.4.1.5 流式返回与路线卡片嵌入

在结果返回阶段，系统采用流式交互方式提升对话自然度。

后端提供 `POST /api/ai/chat/stream` 这一 SSE 流式接口，同时工程中还提供了 WebSocket 会话通道，用于在不同交互模式下承载 AI 消息返回。

前端接收到流式文本后，会持续更新对话消息气泡；当后端返回路线卡片和追问建议时，前端再将其插入到消息流中，形成“文本解释 + 路线卡片 + 下一步建议”的复合式推荐结果。

系统还支持历史会话加载、会话切换、会话删除和上下文续接，这使 AI 助手不再是一次性问答工具，而是可以持续服务用户行前决策的智能助手。

### 5.4.2 天气与景观预测服务实现

#### 5.4.2.1 地理位置解析与天气数据获取

天气与景观预测服务的第一步是确定路线的有效地理位置。

系统优先使用路线轨迹中记录的起点经纬度作为定位依据，因为这类坐标最能反映真实出发区域。

若路线缺少明确起点坐标，系统再根据路线文本地点进行地理解析，从而保证天气服务具备可用位置输入。

位置确定后，系统会进一步获取未来七天天气、当前天气和小时级天气数据，并生成供前端详情页使用的标准化天气对象。

与此同时，系统还会联动天文服务获取日出、日落、月出、月落和月相信息，并结合光污染条件补充夜空环境判断依据。

#### 5.4.2.2 天文数据与随机森林特征构造

在景观预测阶段，天气信息本身并不足以直接给出高质量判断，系统还需要将气象、地理和天文信息进一步组织为模型特征。

从当前工程实现可以看出，系统已经围绕湿度、露点差、云量、风速、降水、温度、海拔等维度构造景观预测所需特征，并按不同景观类型拆分预测逻辑。

其中，云海、雾凇和冰挂等景观更适合通过随机森林进行概率建模，而日出日落和银河等场景则需要结合天文计算、天气条件和夜空环境进行联合判断。

这种“规则与模型结合”的方式，既符合户外景观形成机制的客观特点，也更适合毕业设计阶段的数据基础与工程实现条件。

#### 5.4.2.3 Python Flask 随机森林预测服务调用

按照任务书和开题报告的既有技术路线，系统在景观预测实现中采用 Python Flask 封装随机森林微服务。

后端在完成位置解析和特征构造后，将标准化特征向量传递给 Flask 服务进行推理，Flask 服务完成模型加载、特征校验、概率计算和结果返回。

Java 后端则负责统一接口暴露、结果缓存、异常处理和多景观类型结果整合，最终通过 `GET /api/trails/{id}/landscape-prediction` 向前端返回结构化景观预测结果。

在当前工程基础上，系统已经具备多类景观预测器、预测结果对象、特征快照和人工标签表等支撑内容，这为 Python Flask 微服务形态提供了直接的工程映射基础。

#### 5.4.2.4 结果展示与风险提示

景观预测结果并不只是一组概率数字，而应转化为用户可理解、可操作的决策信息。

因此，系统在前端详情页中会将预测结果组织为景观卡片，展示景观类型、概率等级、推荐观赏时段、影响因素和风险提示。

例如，当云量偏高、风速偏大或露点差不足时，系统会明确提示当前环境不利于特定景观形成；当日出时段与天气条件较优时，则会突出推荐时间窗。

这种展示方式使天气服务与景观预测不再只是信息叠加，而是能够真正服务于户外出行决策。
